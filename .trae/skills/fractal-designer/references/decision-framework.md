本文档提供系统化的设计决策框架，帮助生成高质量的三套设计方案并引导用户做出明智选择。

## 2. 决策点识别方法

### 2.1 设计决策点分类体系

```
设计决策点（Design Decision Points）
│
├── 战略层决策（Strategic）
│   ├── 产品定位决策
│   ├── 目标用户决策
│   ├── 核心价值主张决策
│   └── 竞争差异化决策
│
├── 结构层决策（Structural）
│   ├── 功能模块划分决策
│   ├── 信息架构决策
│   ├── 导航结构决策
│   └── 数据模型决策
│
├── 框架层决策（Skeleton）
│   ├── 交互模式决策
│   ├── 界面布局决策
│   ├── 组件选型决策
│   └── 动效策略决策
│
├── 表层决策（Surface）
│   ├── 视觉风格决策
│   ├── 色彩方案决策
│   ├── 字体排版决策
│   └── 图标插画决策
│
└── 技术支撑决策（Technical - 服务于设计）
    ├── 技术栈匹配决策
    ├── 性能优化策略决策
    ├── 安全方案决策
    └── 可访问性决策
```

### 2.2 决策点识别检查表

在每一层级的设计分析阶段，使用以下问题识别决策点：

#### L0 层级（初步想法）

| # | 识别问题 | 对应决策类型 | 必须询问? |
|---|----------|--------------|-----------|
| L0-01 | 这个产品/功能要解决用户的什么核心问题？ | 战略层 | ✅ 必须 |
| L0-02 | 目标用户是谁？他们的核心特征是什么？ | 战略层 | ✅ 必须 |
| L0-03 | 用户使用这个产品的典型场景是什么？ | 战略层 | ✅ 必须 |
| L0-04 | 用户期望获得什么样的体验感受？ | 战略层 | ✅ 必须 |
| L0-05 | 有哪些时间上的约束或预期？ | 战略层 | ⚠️ 建议 |

#### L1 层级（功能模块）

| # | 识别问题 | 对应决策类型 | 必须询问? |
|---|----------|--------------|-----------|
| L1-01 | 应该将系统划分为哪些主要模块？ | 结构层 | ✅ 必须 |
| L1-02 | 各模块之间的主次关系和依赖关系如何？ | 结构层 | ✅ 必须 |
| L1-03 | 用户最常使用的核心路径是什么？ | 结构层 | ✅ 必须 |
| L1-04 | 模块间的数据如何流转？ | 结构层 | ✅ 必须 |
| L1-05 | 首屏应该展示什么内容给用户？ | 框架层 | ✅ 必须 |

#### L2 层级（子功能）

| # | 识别问题 | 对应决策类型 | 必须询问? |
|---|----------|--------------|-----------|
| L2-01 | 这个子功能的主要操作流程是什么？ | 框架层 | ✅ 必须 |
| L2-02 | 用户完成操作需要几个步骤？可以简化吗？ | 框架层 | ✅ 必须 |
| L2-03 | 操作成功/失败时应该如何反馈用户？ | 框架层 | ✅ 必须 |
| L2-04 | 是否需要支持批量操作？ | 框架层 | ⚠️ 建议 |
| L2-05 | 异常情况如何处理和提示？ | 框架层 | ✅ 必须 |

#### L3 层级（组件）

| # | 识别问题 | 对应决策类型 | 必须询问? |
|---|----------|--------------|-----------|
| L3-01 | 组件的交互状态有哪些？ | 表层 | ✅ 必须 |
| L3-02 | 组件在不同屏幕尺寸下如何表现？ | 表层 | ⚠️ 建议 |
| L3-03 | 组件是否需要动画过渡效果？ | 表层 | ⚠️ 建议 |
| L3-04 | 组件的可访问性要求是什么？ | 技术支撑 | ✅ 必须 |

## 3. 三套方案生成方法论

### 3.1 方案差异化生成算法

```python
def generate_three_solutions(decision_point, context):
    """
    根据决策点和上下文生成三套差异化方案
    
    参数:
        decision_point: 决策点描述
        context: 设计上下文(用户需求、约束等)
    
    返回:
        [solution_a, solution_b, solution_c] 三套方案
    """
    
    solutions = []
    
    # ====== 方案A：保守稳健型 ======
    solution_a = {
        'type': 'conservative',
        'name': f'{decision_point} - 经典方案',
        'design_philosophy': '采用经过验证的成熟模式',
        'approach': select_proven_pattern(decision_point),
        'user_experience': '熟悉可靠、学习成本低',
        'pros': [
            '低风险、高成功率',
            '广泛的社区支持和资料',
            '团队成员容易上手',
            '问题排查经验丰富'
        ],
        'cons': [
            '可能缺乏创新亮点',
            '在某些场景下效率不是最优',
            '可能显得"过时"'
        ],
        'tech_suggestion': recommend_mature_tech(context),
        'best_for': '追求稳定性、团队经验有限、时间紧迫的场景'
    }
    solutions.append(solution_a)
    
    # ====== 方案B：平衡优化型（推荐） ======
    solution_b = {
        'type': 'balanced',
        'name': f'{decision_point} - 平衡方案 (推荐)',
        'design_philosophy': '在成熟与创新之间取得平衡',
        'approach': select_balanced_pattern(decision_point, context),
        'user_experience': '兼顾易用性与现代感',
        'pros': [
            '兼顾稳定性和先进性',
            '较好的长期可维护性',
            '适度的竞争优势',
            '合理的实施成本'
        ],
        'cons': [
            '需要一定的技术评估能力',
            '部分新技术可能存在未知风险',
            '需要更多前期调研'
        ],
        'tech_suggestion': recommend_balanced_tech(context),
        'best_for': '大多数项目的最佳选择、希望适度创新的场景',
        'recommended': True
    }
    solutions.append(solution_b)
    
    # ====== 方案C：前沿创新型 ======
    solution_c = {
        'type': 'innovative',
        'name': f'{decision_point} - 创新方案',
        'design_philosophy': '探索前沿技术与方法论',
        'approach': select_innovative_pattern(decision_point),
        'user_experience': '新颖独特、可能带来惊喜',
        'pros': [
            '显著的差异化优势',
            '可能成为行业标杆',
            '技术前瞻性强',
            '吸引技术人才'
        ],
        'cons': [
            '较高实施风险',
            '生态可能不成熟',
            '学习和调试成本高',
            '可能出现未知问题'
        ],
        'tech_suggestion': recommend_cutting_edge_tech(context),
        'best_for': '追求突破、有充足资源和技术储备的场景'
    }
    solutions.append(solution_c)
    
    return solutions
```

### 3.2 方案质量保证清单

生成三套方案后，必须通过以下质量检查：

| 检查项 | 标准 | 不合格处理 |
|--------|------|------------|
| **差异度** | 三套方案至少在3个维度上有明显差异 | 重新调整方案，增加区分度 |
| **可行性** | 每套方案技术上都是可行的 | 移除不可行方案，替换新方案 |
| **完整性** | 每套方案都包含所有必需要素（理念/体验/优劣势/技术/场景） | 补充缺失要素 |
| **中立性** | 方案描述不带倾向性（除推荐标注外） | 调整措辞，保持客观 |
| **具体性** | 描述足够具体，不是空泛的概念 | 增加具体示例和细节 |

## 4. 多维度评估矩阵

### 4.1 通用评估维度

对每个决策点的三套方案，从以下维度进行评估：

| 维度 | 权重 | 评估标准 | 评分说明 |
|------|------|----------|----------|
| **用户体验** | 25% | 用户使用时的满意度、效率、愉悦感 | 1-5分，5分最优 |
| **实现可行性** | 20% | 技术、资源、时间上的可实现程度 | 1-5分，5分最容易 |
| **可维护性** | 15% | 后续修改、扩展、排错的便利性 | 1-5分，5分最好维护 |
| **扩展性** | 15% | 未来功能迭代的适应能力 | 1-5分，5分最灵活 |
| **风险等级** | 15% | 实施过程中可能遇到的问题和不确定性 | Low/Medium/High |
| **创新价值** | 10% | 相对于竞品的差异化和竞争力 | 1-5分，5分最具创新 |

## 5. AskUserQuestion 使用规范

### 5.1 问题设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **清晰明确** | 问题本身不含歧义 | ✅ "用户登录时支持哪些方式？" <br> ❌ "登录怎么搞？" |
| **选项完整** | 覆盖主要可能性，提供"其他"选项 | 至少3个预设选项 + Other |
| **描述充分** | 每个选项附带简要说明 | 让用户理解选择的含义 |
| **避免诱导** | 不在问题描述中暗示偏好 | ❌ "你不会想用老方案吧？" |
| **推荐标注** | 如有推荐方案，明确标注并说明理由 | "方案B (推荐): 因为..." |

### 5.2 标准问题模板

#### 类型1：单选决策（最常用）
