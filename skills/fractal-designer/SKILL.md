# Fractal Designer - 分形式设计器

## 技能概述

本技能采用**分形递归思想**，从用户的初步想法开始，通过多层级、逐步细化的方式将概念转化为**完善的标准设计文档集**。每个技术分叉点、冲突点、需要决策或有疑问的地方都必须询问用户，确保设计符合用户预期。标准设计文档集可一直细化，直到用户主动停止细化。

## 核心目标：完善的标准设计文档集

### 各文档详细说明

#### 1. 需求类文档

| 文档名称 | 说明 | 核心内容 |
|---------|------|---------|
| **需求规格说明书 (SRS)** | 完整描述系统功能和非功能需求 | 功能需求清单、非功能需求、约束条件、假设依赖 |
| **用户故事/用例文档** | 从用户视角描述功能场景 | 用户故事地图、Use Case、流程图 |
| **需求追踪矩阵** | 需求与后续设计、代码、测试的映射 | RTM表格、需求ID映射 |

#### 2. 设计类文档

| 文档名称 | 说明 | 核心内容 |
|---------|------|---------|
| **架构设计文档** | 系统整体架构、技术选型、模块划分 | 架构图、技术栈选型、模块职责划分 |
| **详细设计文档** | 各模块的具体实现方案 | 类图、时序图、算法设计 |
| **接口设计文档** | API接口定义、协议规范 | 接口清单、请求/响应格式、错误码 |
| **数据库设计文档** | ER图、表结构、索引设计 | ER图、表结构、索引策略 |
| **UI/UX设计文档** | 界面原型、交互设计 | 页面原型、交互流程、视觉规范 |

#### 3. 测试类文档

| 文档名称 | 说明 | 核心内容 |
|---------|------|---------|
| **测试计划** | 测试范围、策略、资源安排 | 测试范围、测试策略、资源分配 |
| **测试用例** | 具体测试场景和预期结果 | 用例编号、输入、步骤、预期结果 |
| **测试报告** | 测试执行结果和问题汇总 | 执行情况、缺陷统计、质量评估 |

#### 4. 部署运维类文档

| 文档名称 | 说明 | 核心内容 |
|---------|------|---------|
| **部署文档** | 部署步骤、环境配置 | 部署架构、环境配置、部署步骤 |
| **运维手册** | 日常运维操作指南 | 监控指标、巡检流程、常见问题 |
| **用户手册** | 最终用户使用指南 | 功能说明、操作指南、FAQ |

#### 5. 项目管理文档

| 文档名称 | 说明 | 核心内容 |
|---------|------|---------|
| **项目计划** | 工期安排、里程碑定义 | WBS、里程碑、资源计划 |
| **风险评估文档** | 识别风险及应对策略 | 风险清单、概率影响评估、应对策略 |

#### 6. 其他重要文档

| 文档名称 | 说明 | 核心内容 |
|---------|------|---------|
| **变更记录** | 需求/设计变更历史 | 变更原因、变更内容、影响评估 |

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 产出文档 |
|------|------|------|---------|
| L0 | 初步想法 | 用户原始想法、概念描述 | 概念草图 |
| L1 | 需求分析 | 功能需求、非功能需求、目标用户 | SRS、用户故事、需求追踪矩阵 |
| L2 | 方案设计 | 架构方案、技术选型、交互流程 | 架构设计文档、接口设计概要 |
| L3 | 详细设计 | 界面设计、API设计、数据结构 | 详细设计文档、接口设计详版、数据库设计 |
| L4 | 实现设计 | 具体实现方案、代码结构 | 前端/后端实现设计、测试计划初稿 |
| L5+ | 持续细化 | 部署设计、运维设计、用户手册等 | 部署文档、运维手册、用户手册等 |

### 递归设计模式（自相似）

每个层级都遵循相同的设计模式：

```
┌─────────────────────────────────────────────────────────────────────┐
│  层级 N 设计模式（自相似）                                            │
├─────────────────────────────────────────────────────────────────────┤
│  1. 层级 N 设计分析 → 2. 识别决策点/分叉点/冲突点/疑问点             │
│  ↓                                                                    │
│  3. 询问用户决策 → 4. 记录用户选择 → 5. 生成该层级设计文档           │
│  ↓                                                                    │
│  6. 识别可产出的标准文档 → 7. 更新标准设计文档集                      │
│  ↓                                                                    │
│  8. 判断是否深入下一层级（递归决策点）                                │
│  ↓ 是（递归）                                                         │
│  层级 N+1 设计（重复上述模式）                                        │
└─────────────────────────────────────────────────────────────────────┘
```

### 文档生成时机

```
层级完成时 → 生成该层级对应的标准设计文档 → 追加到标准设计文档集

L0 完成后 → 生成《概念草图》
L1 完成后 → 生成《需求规格说明书》《用户故事文档》《需求追踪矩阵》
L2 完成后 → 生成《架构设计文档》《接口设计概要》
L3 完成后 → 生成《详细设计文档》《接口设计详版》《数据库设计文档》
L4 完成后 → 生成《前端实现设计》《后端实现设计》《测试计划》
L5+ 完成后 → 生成《部署文档》《运维手册》《用户手册》等
```

## 详细步骤

### 第一阶段：L0 初步想法（概念级）

#### 1.1 收集用户初步想法
- 请用户描述初步想法或概念
- 记录用户的原始描述
- 确认理解是否正确

#### 1.2 识别 L0 决策点
- 核心目标是什么？
- 主要使用场景？
- 目标用户群体？
- 时间预期？

#### 1.3 询问用户决策
使用 AskUserQuestion 工具询问用户关于 L0 决策点的选择。

#### 1.4 生成 L0 设计文档
生成《概念草图》并保存到：`docs/design/{项目名}/fractal-design-{YYYYMMDD}-L0-concept.md`

#### 1.5 询问是否继续细化
询问用户是否进入 L1 需求分析阶段。

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似设计模式
- **必须**在每个技术分叉点、冲突点、需要决策或有疑问的地方使用 AskUserQuestion 工具询问用户
- **必须**在每一步设计完成后立即生成标准设计文档，防止内容丢失
- **必须**记录所有用户决策，保存到决策记录文件
- **必须**在每个层级完成后询问用户是否继续细化（递归决策点）
- **必须**用户控制细化深度，当用户表示不再继续时停止细化
- **不要**自作主张做决策，所有技术选择都必须询问用户
- 完成的设计工作写到 `docs/achievement/achievement-{日期}.md` 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 技术选型（框架、库、工具等）
2. 架构方案选择
3. 功能优先级确定
4. UI/UX 设计方案
5. 数据结构设计
6. API 设计方案
7. 部署方案
8. 运维方案
9. 任何有多种可选方案的情况
10. 任何不确定用户意图的情况
11. 任何可能存在冲突的设计点

### 递归终止条件
- **用户主动要求停止**（最优先）
- 设计已达到用户需要的详细程度
- 用户表示不需要更多文档

## 何时使用

- 用户有初步想法需要细化
- 新项目启动前的设计阶段
- 新功能开发前的设计阶段
- 需要系统化梳理设计思路时
- 需要明确技术选型时
- 需要解决设计冲突时
- 需要生成完整标准设计文档集时

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种选择的利弊
- 给予用户充分的选择权，不要预设答案
- 递归过程中保持上下文连贯性
- **每一步都要更新决策记录文件**，防止内容丢失
- 记录所有决策过程，便于后续追溯
- 如果用户对某个领域不熟悉，要提供详细的选项说明
- 如果遇到技术分叉点、冲突点、决策或有疑问的地方，**必须**使用 AskUserQuestion 工具问用户，**不要**自作主张
- **用户有权在任何时候停止细化**，尊重用户的时间安排和需求
