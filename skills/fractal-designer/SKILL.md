# Fractal Designer - 分形式设计器

## 技能概述

本技能采用**分形递归思想** + **横向拆分思想**，从用户的初步想法开始，通过多层级、逐步细化的方式将概念转化为**完善的标准设计文档集**。
- **纵向**：按层级逐步细化（L0 → L1 → L2 → L3 → L4 → L5+）
- **横向**：每个层级按功能模块/分类拆分，同级并行执行
- **文档**：单一总文档，层级嵌套保存
- **验证**：任务完成后进行正向和反向验证

每个技术分叉点、冲突点、需要决策或有疑问的地方都必须询问用户，确保设计符合用户预期。标准设计文档集可一直细化，直到用户主动停止细化。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 | 横向拆分 |
|------|------|------|--------|----------|
| L0 | 初步想法 | 用户原始想法、概念描述 | 概念级 | 无（单独执行） |
| L1 | 功能模块级 | 按功能模块拆分 | 宏观级 | 功能模块A、功能模块B、功能模块C... |
| L2 | 子功能级 | 功能模块下的子功能 | 中观级 | 子功能1、子功能2、子功能3... |
| L3 | 组件级 | 具体组件/接口设计 | 微观级 | 组件1、组件2、组件3... |
| L4 | 实现级 | 具体实现方案 | 原子级（可选） | 实现项1、实现项2、实现项3... |
| L5+ | 持续细化 | 部署、运维、用户手册等 | 持续级（可选） | 无 |

### 递归设计模式（自相似）

每个层级都遵循相同的设计模式：

```
┌─────────────────────────────────────────────────────────────────────┐
│  层级 N 设计模式（自相似）                                            │
├─────────────────────────────────────────────────────────────────────┤
│  1. 层级 N 设计分析 → 2. 识别决策点/分叉点/冲突点/疑问点             │
│  ↓                                                                    │
│  3. 询问用户决策 → 4. 记录用户选择 → 5. 推荐横向拆分方案             │
│  ↓ 询问用户确认（如有）                                               │
│  6. 保存当前内容到总文档 → 7. 判断是否深入下一层级（递归决策点）      │
│  ↓ 是（递归）                                                         │
│  层级 N+1 设计（重复上述模式）                                        │
└─────────────────────────────────────────────────────────────────────┘
```

### 横向并行执行模式

```
L0（单独执行）
    ↓
L1（按功能模块拆分，多个子Agent并行）
    ├─ 功能模块A（子Agent 1）
    ├─ 功能模块B（子Agent 2）
    ├─ 功能模块C（子Agent 3）
    └─ ...
    ↓
总结、检验、修改总文档
    ↓
L2（按子功能拆分，多个子Agent并行）
    ↓
总结、检验、修改总文档
    ↓
...
    ↓
总结、正向+反向验证
```

# 总结与验证

## 正向验证（设计 → 文档）
逐条检查每条设计，确保都有明确的文档依据：
- [ ] 验证项1
- [ ] 验证项2

## 反向验证（文档 → 设计）
遍历所有文档，确保文档中的每个要求都在设计中体现：
- [ ] 验证项1
- [ ] 验证项2

## 正确性验证
检查每个层级的设计是否正确：
- [ ] L0 正确性
- [ ] L1 正确性
- [ ] L2 正确性
- [ ] ...

## 一致性验证
确保各层级之间的内容一致：
- [ ] L0 → L1 一致性
- [ ] L1 → L2 一致性
- [ ] ...

## 标准设计文档索引
[文档索引]

### 文档保存时机
- **任务开始时**：创建任务总文档，路径：`docs/design/fractal-design-{YYYYMMDD}.md`
- **L0 完成后**：写入 L0 内容，询问 L1 横向拆分方案
- **每个 L1 模块完成后**：写入对应 L1 内容，询问 L2 横向拆分方案
- **每个层级全部完成后**：总结、检验、修改总文档
- **任务完成后**：进行正向+反向+正确性+一致性验证，写入验证结果

## 详细步骤

### 第一阶段：L0 初步想法（概念级）- 单独执行

#### 1.1 创建任务总文档
创建单一总文档，路径：`docs/design/fractal-design-{YYYYMMDD}.md`

#### 1.2 收集用户初步想法
- 请用户描述初步想法或概念
- 记录用户的原始描述
- 确认理解是否正确

#### 1.3 识别 L0 决策点
- 核心目标是什么？
- 主要使用场景？
- 目标用户群体？
- 时间预期？

#### 1.4 询问用户决策
使用 AskUserQuestion 工具询问用户关于 L0 决策点的选择。

#### 1.5 推荐 L1 横向拆分方案
向用户推荐 L1 层级的横向拆分方案（按功能模块）：
| 序号 | L1 功能模块 | 说明 | 建议优先级 |
|------|------------|------|-----------|
| 1 | 功能模块A | [说明] | 高 |
| 2 | 功能模块B | [说明] | 中 |
| 3 | 功能模块C | [说明] | 中 |

**重要**：必须使用 AskUserQuestion 工具询问用户确认/调整拆分方案！

#### 1.6 写入 L0 内容到总文档
将 L0 所有内容写入总文档。

#### 1.7 询问是否继续细化（进入 L1）

---

### 第二阶段：L1 功能模块级设计（宏观级）- 同级并行

#### 2.1 L1 横向并行执行
根据用户确认的 L1 拆分方案，调用多个子Agent并行执行：
- 子Agent 1：功能模块A
- 子Agent 2：功能模块B
- 子Agent 3：功能模块C
- ...

#### 2.2 每个 L1 模块的设计流程（自相似）

```
┌─────────────────────────────────────────────────────┐
│ L1 模块设计（应用自相似模式）                        │
├─────────────────────────────────────────────────────┤
│ 1. L1 设计分析 → 2. 识别决策点 → 3. 询问用户         │
│ ↓                                                     │
│ 4. 记录用户选择 → 5. 推荐 L2 拆分方案 → 6. 询问用户 │
│ ↓                                                     │
│ 7. 写入总文档 → 8. 是否深入 L2？                      │
└─────────────────────────────────────────────────────┘
```

#### 2.3 L1 设计内容
- 功能模块概述
- 核心功能分析
- 技术栈初步确认
- 与其他模块的关系

#### 2.4 推荐 L2 横向拆分方案
每个 L1 模块完成后，向用户推荐 L2 层级的横向拆分方案（按子功能）。

**重要**：必须使用 AskUserQuestion 工具询问用户确认/调整拆分方案！

#### 2.5 汇总所有 L1 模块结果
主Agent整合所有子Agent的 L1 结果，总结、检验、修改总文档。

#### 2.6 进入 L2 阶段（或跳过）

---

### 第三阶段：L2 子功能级设计（中观级）- 同级并行

#### 3.1 L2 横向并行执行
对于每个 L1 模块下的 L2 子功能，调用多个子Agent并行执行。

#### 3.2 L2 设计流程（继续自相似模式）
同 L1 模式，推荐 L3 横向拆分方案（按组件）。

#### 3.3 汇总所有 L2 子功能结果
主Agent整合所有子Agent的 L2 结果，总结、检验、修改总文档。

---

### 第四阶段：L3 组件级设计（微观级）- 同级并行

#### 4.1 L3 横向并行执行
对于每个 L2 子功能下的 L3 组件，调用多个子Agent并行执行。

#### 4.2 L3 设计流程（继续自相似模式）
同 L1 模式，可选推荐 L4 横向拆分方案（按实现项）。

#### 4.3 汇总所有 L3 组件结果
主Agent整合所有子Agent的 L3 结果，总结、检验、修改总文档。

---

### 第五阶段：L4 实现级设计（原子级）- 可选深度递归

#### 5.1 L4 横向并行执行（可选）
对于每个 L3 组件下的 L4 实现项，调用多个子Agent并行执行。

#### 5.2 L4 设计流程（继续自相似模式）
同 L1 模式，递归终止或进入 L5+。

#### 5.3 汇总所有 L4 实现项结果
主Agent整合所有子Agent的 L4 结果，总结、检验、修改总文档。

---

### 第六阶段：L5+ 持续细化（部署运维等）- 可选

#### 6.1 L5+ 设计内容
- 部署设计
- 运维设计
- 用户手册
- 项目计划
- 风险评估
- 应急预案

#### 6.2 写入 L5+ 内容到总文档

---

### 第七阶段：总结与验证

#### 7.1 正向验证（设计 → 文档）
逐条检查每条设计，确保都有明确的文档依据：
- 文档名称
- 章节/行号
- 具体引用内容

#### 7.2 反向验证（文档 → 设计）
遍历所有文档，确保文档中的每个要求都在设计中体现：
- 建立文档内容清单
- 与设计列表逐项比对
- 标记遗漏项
- 补充缺失设计

#### 7.3 正确性验证
检查每个层级的设计是否正确：
- L0 正确性
- L1 正确性
- L2 正确性
- ...

#### 7.4 一致性验证
确保各层级之间的内容一致：
- L0 → L1 一致性
- L1 → L2 一致性
- ...

#### 7.5 生成标准设计文档索引
创建文档索引，列出所有已生成的标准设计文档。

#### 7.6 写入验证结果到总文档
将所有验证结果、文档索引写入总文档。

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似设计模式
- **必须**在每个层级开始时推荐横向拆分方案，使用 AskUserQuestion 询问用户确认
- **必须**调用多个子Agent并行执行同级任务
- **必须**每个层级全部完成后，总结、检验、修改总文档
- **必须**任务完成后进行正向+反向+正确性+一致性验证
- **必须**使用单一总文档，所有层级内容嵌套保存
- **必须**在每个技术分叉点、冲突点、需要决策或有疑问的地方使用 AskUserQuestion 工具询问用户
- **必须**用户控制细化深度，当用户表示不再继续时停止细化
- **不要**自作主张做决策，所有技术选择都必须询问用户
- 完成的设计工作写到 `docs/achievement/achievement-{日期}.md` 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 技术选型（框架、库、工具等）
2. 架构方案选择
3. 功能优先级确定
4. UI/UX 设计方案
5. 数据结构设计
6. API 设计方案
7. 横向拆分方案确认
8. 任何有多种可选方案的情况
9. 任何不确定用户意图的情况
10. 任何可能存在冲突的设计点

### 递归终止条件
- **用户主动要求停止**（最优先）
- 设计已达到用户需要的详细程度
- 用户表示不需要更多文档

## 何时使用

- 用户有初步想法需要细化
- 新项目启动前的设计阶段
- 新功能开发前的设计阶段
- 需要系统化梳理设计思路时
- 需要明确技术选型时
- 需要解决设计冲突时
- 需要生成完整标准设计文档集时

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种选择的利弊
- 每个层级都要推荐清晰的横向拆分方案
- 给予用户充分的选择权，不要预设答案
- 同级任务并行执行，提高效率
- 递归过程中保持上下文连贯性
- **每一步都要更新总文档**，防止内容丢失
- 单一总文档，层级嵌套保存，便于追溯
- 记录所有决策过程，便于后续追溯
- 如果用户对某个领域不熟悉，要提供详细的选项说明
- 如果遇到技术分叉点、冲突点、决策或有疑问的地方，**必须**使用 AskUserQuestion 工具问用户，**不要**自作主张
- **用户有权在任何时候停止细化**，尊重用户的时间安排和需求
