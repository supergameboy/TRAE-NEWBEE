# Fractal Designer - 分形式设计器

## 技能概述

本技能采用**分形递归思想**，从用户的初步想法开始，通过多层级、逐步细化的方式将概念转化为可落地的设计方案。每个技术分叉点、冲突点、需要决策或有疑问的地方都必须询问用户，确保设计符合用户预期。每一步设计内容都会保存到文档，防止内容丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 |
|------|------|------|--------|
| L0 | 初步想法 | 用户原始想法、概念描述 | 概念级 |
| L1 | 需求分析 | 功能需求、非功能需求、目标用户 | 宏观级 |
| L2 | 方案设计 | 架构方案、技术选型、交互流程 | 中观级 |
| L3 | 详细设计 | 界面设计、API设计、数据结构 | 微观级 |
| L4 | 可落地实现 | 具体实现方案、代码结构、交付物 | 原子级（可选） |

### 递归设计模式（自相似）

每个层级都遵循相同的设计模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 设计模式（自相似）                                      │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 设计分析 → 2. 识别决策点/分叉点/冲突点/疑问点       │
│  ↓                                                              │
│  3. 询问用户决策 → 4. 记录用户选择 → 5. 保存当前设计文档       │
│  ↓                                                              │
│  6. 判断是否深入下一层级（递归决策点）                          │
│  ↓ 是（递归）                                                   │
│  层级 N+1 设计（重复上述模式）                                  │
└───────────────────────────────────────────────────────────────┘
```

### 文档保存时机
- **每个层级完成后**：立即保存当前层级的设计文档
- **每次用户决策后**：立即更新文档中的决策记录
- **发现新的分叉点时**：立即记录到文档中
- **递归进入下一层级前**：确保当前层级文档已保存

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似设计模式
- **必须**在每个技术分叉点、冲突点、需要决策或有疑问的地方使用 AskUserQuestion 工具询问用户
- **必须**在每一步设计完成后立即保存文档，防止内容丢失
- **必须**记录所有用户决策，保存到设计文档的决策记录中
- **必须**在每个层级验证后询问用户是否深入下一层级（递归决策点）
- **不要**自作主张做决策，所有技术选择都必须询问用户
- **不要**跳过层级，必须按照 L0→L1→L2→L3→L4 的顺序推进
- 完成的设计工作写到 docs/achievement/achievement-{日期}.md 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 技术选型（框架、库、工具等）
2. 架构方案选择
3. 功能优先级确定
4. UI/UX 设计方案
5. 数据结构设计
6. API 设计方案
7. 任何有多种可选方案的情况
8. 任何不确定用户意图的情况
9. 任何可能存在冲突的设计点

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L4 可落地实现（最细颗粒度）
- 用户主动要求停止
- 设计已达到可落地程度，无需继续细化

## 何时使用

- 用户有初步想法需要细化
- 新项目启动前的设计阶段
- 新功能开发前的设计阶段
- 需要系统化梳理设计思路时
- 需要明确技术选型时
- 需要解决设计冲突时

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种选择的利弊
- 给予用户充分的选择权，不要预设答案
- 递归过程中保持上下文连贯性
- **每一步都要保存文档**，防止内容丢失
- 记录所有决策过程，便于后续追溯
- 如果用户对某个领域不熟悉，要提供详细的选项说明
- 如果遇到技术分叉点、冲突点、决策或有疑问的地方，**必须**使用 AskUserQuestion 工具问用户，**不要**自作主张
