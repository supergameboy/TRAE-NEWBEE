# Task Scheduler Fractal - 分形式项目任务规划器

## 技能概述

本技能采用**分形思想**，通过多层级、逐步细化的方式制定详细、可执行的项目任务计划文档。针对文档过多的情况，采用分批次读取策略，从宏观到微观逐步深化，确保任务规划的完整性和精确性。

## 分形工作流程

```
第一层级：文档概览 → 第二层级：框架规划 → 第三层级：细化深化 → 第四层级：验证确认
```

## 详细步骤

### 第一层级：文档概览阶段（Level 1 - 宏观）

#### 1.1 分批次读取全部文档
使用 Search Agent 分批次读取 docs 目录下的所有文档，避免一次性处理过多文档造成的信息过载。

**批次划分策略：**
- 批次 1：参考文档索引.md、项目架构说明文档.md
- 批次 2：核心功能设计文档（按模块分组）
- 批次 3：技术实现文档（前端/后端/数据库等）
- 批次 4：其他辅助文档

#### 1.2 生成文档情况报告
每批次读取完成后，生成该批次的文档情况报告，包含：
- 文档列表及文件路径
- 文档类型分类
- 文档颗粒度
- 文档简介
- 文档内容摘要
- 文档篇幅评估
- 文档间关联关系初步分析

#### 1.3 合并生成总文档概览报告
汇总所有批次的报告，生成完整的文档概览，包含：
- 文档总量统计
- 文档分类矩阵
- 关键文档识别
- 文档依赖关系图

#### 1.4 生成阶段任务计划文档
生成L1级的任务计划文档，保存到 `docs/task-plans/` 目录，文件名格式为 `task-plan-{时间}.md`。

**注意：**
  **禁止**“继续读取更多的关键文档”，仅根据Search Agent生成的文档概览生成任务计划，需要任何信息必须调用子Agent读取并分析。

### 第二层级：任务框架规划阶段（Level 2 - 中观）

#### 2.1 读取核心文档
从文档概览中识别并读取核心文档，包括：
- 项目架构说明文档.md
- 主要功能模块设计文档
- 技术选型文档

#### 2.2 识别任务模块
从核心文档中提取主要功能模块和技术任务，建立任务框架：

| 模块 | 任务来源 | 颗粒度标记 |
|------|---------|-----------|
| 模块 A | 文档名.md - 第X章 | L2（框架级） |
| 模块 B | 文档名.md - 第Y章 | L2（框架级） |

**颗粒度定义：**
- **L0（项目级）**：整个项目
- **L1（阶段级）**：开发阶段
- **L2（模块级）**：功能模块
- **L3（任务级）**：具体任务
- **L4（子任务级）**：任务分解

#### 2.3 建立任务框架文档
写出框架级的任务计划文档，包含：
- 项目总体目标
- 主要阶段划分
- 核心模块识别
- A/B组初步职责划分
- 颗粒度标记（L2）

#### 2.4 生成阶段任务计划文档
修改L1级的任务计划文档，添加L2级任务的框架。

**注意：**
此阶段主Agent仅读取核心文档，其余文档必须使用子Agent读取并分析。

### 第三层级：细化深化阶段（Level 3 - 微观）

#### 3.1 分批次读取细化文档
按照以下顺序分批次读取剩余文档：
- 批次 1：各模块的详细设计文档
- 批次 2：技术实现细节文档
- 批次 3：UI/UX 设计文档
- 批次 4：测试计划文档

#### 3.2 逐批次细化任务框架
每批次读取后，立即细化任务框架：
- 将 L2 模块级任务细化为 L3 任务级
- 补充任务描述、技术标准、验收标准
- 标记任务依赖关系
- 分配 A/B 组职责
- 设定时间估算

**注意：**
此阶段主Agent仅读取细化文档，其余文档必须使用子Agent读取并分析。

#### 3.3 递归细化（可选）
根据需要，可将 L3 任务进一步细化为 L4 子任务级，形成分形结构。

#### 3.4 生成阶段任务计划文档
修改L1级的任务计划文档，添加L3级任务的框架。

### 第四层级：验证确认阶段（Level 4 - 验证）

#### 4.1 任务到文档的双向验证
**正向验证（任务 → 文档）：**
逐条检查每条任务规划，确保都有明确的文档依据，包含：
- 文档名称
- 章节/行号
- 具体引用内容

**逆向验证（文档 → 任务）：**
遍历所有文档，确保文档中的每个实现要求都在任务文档中体现：
- 建立文档内容清单
- 与任务列表逐项比对
- 标记遗漏项
- 补充缺失任务

#### 4.2 完整性检查清单
- [ ] 所有核心模块都有任务规划
- [ ] 所有技术要求都在任务中体现
- [ ] 每条任务都有文档来源标注
- [ ] A/B 组职责清晰无重叠
- [ ] 任务依赖关系明确
- [ ] 时间规划合理
- [ ] 交付物定义清晰
- [ ] 验收标准明确可衡量

#### 4.3 生成最终任务计划文档
完成验证后，生成完整的任务计划文档，保存到 `docs/task-plans/` 目录。

## 颗粒度标记规则

| 级别 | 名称 | 说明 | 示例 |
|------|------|------|------|
| L0 | 项目级 | 整个项目 | AI小说生成器项目 |
| L1 | 阶段级 | 开发阶段 | 项目初始化阶段 |
| L2 | 模块级 | 功能模块 | 用户认证模块 |

| L4 | 子任务级 | 任务分解 | 编写登录接口单元测试 |

## 关键规则

### 主Agent规则

- 必须严格按照分形层级逐步推进，不得跳级
- 只读取当前层级的文档，不读取其他层级的文档
- 不要完整读取所有文档，而是使用Search Agent读取相关文档
- 当某个层级超过3个文档，必须分批次调用子Agent读取并分析，你负责整合所有批次的结果
- 每层级读取文档后，必须生成该层级的报告，并保存到文件
- 必须完成正向和逆向双向验证
- 确保颗粒度标记清晰准确
- 合理规划任务时间，确保A/B组负载均衡

### Search Agent规则
- Search Agent只负责探索和报告，不制定计划
- 报告要客观、准确、详细，包含文档章节和行号
- 按批次读取，每批次完成后立即报告

### 子Agent规则
- 调用子Agent时，必须明确告诉子Agent的身份
- 必须让子Agent复述：我只能提出方案，不能独自测试

## 何时使用

- 项目启动初期，制定总体开发计划
- 新功能开发前，制定详细实施计划
- 项目里程碑后，制定下一阶段计划
- 团队分工调整时，重新规划任务分配
- 文档数量较多，需要系统化梳理时
- **每一次进行重大修改、更新或复杂任务的执行阶段时**

## 注意事项

- 不要删除代码，而是注释掉
- 完成的工作写到 docs/achievement/achievement-{日期}.md 中
- 未完成的工作写到 docs/todo.md 中
- A/B组任务边界要清晰，避免相互阻塞
- 建立Mock数据机制，确保前端可独立开发
- 每批次读取后及时更新任务框架，不要积压
- 验证阶段要细致，确保无遗漏
