# 头脑风暴技能

## 技能概述

本技能用于在处理问题、BUG时调用，在进行重大修改、更新、复杂任务时进行头脑风暴，通过系统性地探索代码库、收集各Agent的专业意见，最终整合出多个可行方案供用户选择。  

## 工作流程

### 流程图

```
1. 明确任务目标 → 2. 使用Search Agent按功能区域探索 → 3. 收集探索报告 → 4. 各Agent阅读报告并提出方案 → 5. 主Agent整合方案汇总 → 6. 向用户展示方案 → 7. 用户选择方案
```

## 详细步骤

### 1. 明确任务目标
- 与用户进行结构化沟通，获取关于当前待解决问题或任务目标的详细信息，包括具体需求、预期成果及优先级
- 确定本次头脑风暴的精确范围，明确定义需要重点关注的功能区域和排除范围
- 建立任务成功的量化标准和验收条件

### 2. 按功能区域批量调用Search Agent(内置的可调用的智能体，执行检索文件和查看文件的任务)
根据项目结构，按以下功能区域分别调用内置的Search Agent执行检索与文件查看任务：

#### 功能区域划分：
- **后端架构**：包括但不限于：路由定义与配置、服务层实现、数据访问层/仓库模式、中间件配置与实现、数据库连接与操作、API接口设计与实现
- **前端架构**：包括但不限于：组件层次结构、页面布局与实现、路由配置与导航、状态管理策略与实现、API服务封装与调用、前端构建流程
- **数据库**：包括但不限于：表结构设计、索引策略、数据迁移脚本、常用查询语句、事务管理、数据备份与恢复机制
- **配置与工具**：包括但不限于：环境配置文件、构建工具配置、依赖管理、代码质量工具、部署脚本、CI/CD配置
- **项目其他相关**：包括但不限于：项目文档、API文档、代码注释规范、用户反馈记录、版本历史、已知问题清单

**注：根据项目实际需求，可动态添加其他功能区域，所有功能区域划分以项目实际需求为最终依据**

每个Search Agent需返回探索报告，包括但不限于：
- 该功能区域的核心功能与实现概述（200-300字）
- 关键文件列表（包含文件路径、文件名及简要作用说明）
- 现有架构关系图（如无可提供结构化的关系描述）
- 与当前任务之间的关系分析，包括：
  - 任务目标与该功能区域的直接关联点
  - 该功能区域与其他功能区域的交互方式和依赖关系
  - 该功能区域与系统其他核心组件的接口定义和数据流向

### 3. 收集探索报告汇总
- 主Agent系统地收集所有Search Agent提交的探索报告
- 对报告进行去重、整合与标准化处理，生成一份完整的代码库状况综合报告
- 确保报告包含各功能区域的交叉引用和整体架构视图

### 4. 各专业Agent阅读报告并提出方案
- 系统读取可用Agent列表及其专业领域信息
- 根据任务类型和性质，智能选择并调用所有相关专业Agent
- 针对同一个任务，可以调用多个Agent多个角度提出方案

每个专业Agent需完成以下任务：
- 全面阅读并分析完整的代码库状况报告
- 基于自身专业领域和职责范围
- 提出3个具体、可实施的解决方案，分别是快速解决问题的方案、均衡的方案、长远计划但复杂的方案。
- 每个方案必须包含：详细实现步骤、技术选型建议、潜在风险评估及缓解措施、预计工作量评估
- 严格禁止在方案阶段直接修改任何代码

### 5. 主Agent整合方案汇总
- 主Agent收集所有专业Agent提交的解决方案
- 将方案整理成结构化的方案汇总文档，包含统一的格式和分类
- 对每个方案进行多维度评估：
  - 明确标注方案提出者（具体Agent名称）
  - 客观分析方案的技术优点和创新点
  - 识别方案的潜在缺点和局限性
  - 基于可行性、风险、成本和与目标的契合度给出0-10分的推荐指数

### 6. 向用户展示方案
- 以清晰、专业的格式向用户展示所有方案汇总
- 提供方案间的关键指标对比分析（表格形式）
- 详细说明各方案的适用场景、限制条件和预期效果
- 解释推荐指数的评估依据和各维度权重
- 使用**AskUserQuestion**工具提问并等待用户对方案做出选择
### 7. 用户选择方案
- 记录用户的最终决策及选择理由
- 确认用户是否需要对选中方案进行调整或组合
- 保存决策记录作为后续实施阶段的依据
## 关键规则

### 主Agent运行规则
- 必须按照项目功能区域进行系统性规划，全面调用Search Agent执行信息检索任务，确保覆盖所有相关领域，不得有任何功能模块或关键信息的遗漏
- 明确要求Search Agent返回的报告需包含完整的信息来源、详细的分析过程、准确的数据支持及明确的结论，确保信息的完整性、精确性和可靠性
- 在整合各Search Agent提供的方案时，必须保持完全客观中立的立场，对所有方案进行同等权重的评估，不得因个人偏好或先入为主的观念偏向任何特定方案
- 向用户展示最终整合方案时，需采用结构化呈现方式，使用清晰的逻辑框架、简洁的语言表达和直观的可视化元素（如必要），确保方案内容易于用户理解和评估

### 子Agent协作规则
- 调用子Agent执行任务时，必须在初始指令中明确告知子Agent的具体身份标识及其对应的职责范围，确保子Agent清楚自身定位
- 每次调用子Agent时，必须要求子Agent在开始执行任务前复述确认以下内容："我只能提出方案建议，不能独自进行实际测试或执行操作"，以确保子Agent明确自身权限边界
- 子Agent在提出任何方案建议之前，必须完整阅读并充分理解项目提供的代码库状况报告，包括但不限于代码结构、现有功能实现、已知问题及技术约束等关键信息，确保方案建议具备现实可行性

### Search Agent规则
- Search Agent只负责探索和报告，不提出方案
- 报告要客观、准确、详细

## 何时使用

本技能应在以下场景中被调用：

### 触发条件
1. **用户提出问题或报告BUG时** - 当用户描述功能异常、错误行为或提出疑问时
2. **重大修改、系统更新或复杂任务时** - 当需要进行架构调整、大规模重构或复杂功能开发时

### 触发关键词
当用户输入包含以下关键词时，应优先考虑调用本技能：
- **问题类**：问题、BUG、错误、异常、故障
- **建议类**：应该、可以、可能、需要
- **修改类**：修改、更新、调整、重构、改进

### 具体场景
- 进行重大功能修改
- 架构重构
- 复杂新功能开发
- 技术选型决策
- 其他需要多方案对比的场景
