# 经验文件格式详细规范

## 标准四列格式详解

### 完整模板

```markdown
# 经验总结 - {YYYY-MM-DD} - {技能名称}

## 基本信息
- 技能名称：{技能名，如 bug-hunter-fractal}
- 执行时间：{开始时间 ~ 结束时间，如 2026-05-12 14:30 ~ 15:45}
- 项目上下文：{简要描述本次任务的上下文}

## 关键步骤
1. {具体步骤1}
2. {具体步骤2}
3. {具体步骤3}

## 坑是什么 → 怎么避 → 相关工具/文件 → 统一原则

| 坑是什么 | 怎么避 | 相关工具/文件 | 统一原则 |
|----------|--------|--------------|----------|
| 具体描述坑 | 具体如何避免 | 相关文件路径/工具 | 提炼的通用原则 |

## 优化建议
- {建议1}
- {建议2}

## 本次可沉淀结论
- {结论1}
- {结论2}
```

## 四列内容填写指南

### 1. 坑是什么

**要求**：具体、可重现的问题描述，不说空话。

**好例子**：
- "习惯性以为后端默认端口是3000，实际配置是8080"
- "前端状态更新函数只处理已有数据，新数据被忽略"
- "服务A输出字段为userInfo，但服务B只读取user"

**坏例子**：
- "有个bug"（太笼统）
- "代码有问题"（不具体）
- "性能不好"（无细节）

### 2. 怎么避

**要求**：具体可操作的步骤，不是空泛的建议。

**好例子**：
- "后端端口在 `src/config/app.js` 的 PORT 配置，默认8080"
- "在状态更新函数中添加 else 分支：`state.items.push(newItem)`"
- "服务B修改为兼容读取：`userInfo ?? user`"

**坏例子**：
- "仔细点"（不具体）
- "检查代码"（无操作步骤）
- "注意一下"（不可执行）

### 3. 相关工具/文件

**要求**：具体的文件路径或工具名称，用反引号包裹。

**好例子**：
- "`src/config/app.js`"
- "`src/stores/useAppStore.js` L150"
- "WebSearch、AskUserQuestion"

**坏例子**：
- "某个文件"（不具体）
- "相关文件"（无路径）
- "工具"（无名称）

### 4. 统一原则

**要求**：从具体问题中提炼的通用规则，可应用于其他场景。

**好例子**：
- "环境配置是运行前提——端口、依赖、数据库、环境变量必须逐一确认后再启动服务"
- "数据变更必须双向同步——新增字段/事件/映射时，定义端和消费端必须同步更新"
- "状态管理要考虑完整生命周期——更新、新增、删除都要有对应处理"

**坏例子**：
- "要注意"（不提炼）
- "这个很重要"（空话）
- "记住"（无原则）

## 完整示例（通用示例）

```markdown
# 经验总结 - 2026-05-12 - task-scheduler-fractal

## 基本信息
- 技能名称：task-scheduler-fractal
- 执行时间：2026-05-12 02:30 ~ 03:15
- 项目上下文：系统性能优化从设计稿转实施计划

## 关键步骤
1. 先基于已确认的设计稿锁定实施节奏，再用 AskUserQuestion 明确本轮优先级
2. 技能中提到的模板文件如果仓库不存在，不要照搬；改读 `docs/plans/` 里的既有计划文档
3. 总规划文档只负责阶段、依赖、验收和参考资料，阶段文档再展开到可执行任务粒度
4. 优先级最高的阶段要把"最短路径收益"拆成模块化任务

## 坑是什么 → 怎么避 → 相关工具/文件 → 统一原则

| 坑是什么 | 怎么避 | 相关工具/文件 | 统一原则 |
|----------|--------|--------------|----------|
| 技能文档里的模板路径在仓库中不存在，直接套用会阻塞任务规划 | 先用 Glob/LS/Read 验证仓库真实模板来源，优先复用项目现有计划文档风格 | `docs/plans/plan-202604-v1.md`、`AskUserQuestion` | 以仓库事实为准，不以技能示例路径为准 |
| 设计稿很完整，但实施计划如果一次展开过多阶段，会影响近期执行效率 | 先确认节奏，再优先展开最短路径的阶段；长期阶段在总规划中占位即可 | `docs/designs/optimization-spec.md` | 先拆近期要做的，不把长期路线一次性写成不可执行大计划 |
| 任务描述只有标题，没有依赖和接口信息，后续实现容易返工 | 在阶段文档里同步写任务依赖、数据传递、对接接口和验收标准 | `docs/plans/plan-202605-phase1.md` | 任务计划必须可直接指导实现，而不只是任务列表 |

## 优化建议
- 大型优化项目的任务规划优先采用"两层输出"：总规划 + 当前优先阶段计划
- 决策点必须在写计划前确认，否则阶段划分和任务粒度容易失焦
- 文档一旦写入 `docs/plans/`，同步更新 achievement 与 todo，保证执行闭环

## 本次可沉淀结论
- 对大型改造项目，最有效的计划方式不是一次性展开全部阶段，而是先锚定优先阶段，再把该阶段拆到可直接实现的粒度
```

## 另一个示例（bug排查）

```markdown
# 经验总结 - 2026-05-10 - bug-hunter-fractal

## 基本信息
- 技能名称：bug-hunter-fractal
- 执行时间：2026-05-10 14:45 ~ 15:31
- 项目上下文：面板数据不更新Bug排查

## 关键步骤
1. L0 现象收集：读取日志，确认"操作后界面不更新，刷新后才出现"
2. L1 领域判断：数据库正确保存 → 问题在后端到前端的数据推送链路
3. L2 模块定位：锁定响应构建器（构建更新数据）和服务A（输出格式）
4. L3 文件定位：`ResponseBuilder.js` L617-L657、`service-a.js`、`store.js`
5. L4 代码定位：`data.id` 为 undefined 因为服务A输出 `dataInfo` 而非 `data`
6. L5 根因分析（五问法）：确认响应构建器只检查 `data` 不检查 `dataInfo`
7. 横向检查所有服务：对比多个服务输出字段与响应构建器读取字段

## 坑是什么 → 怎么避 → 相关工具/文件 → 统一原则

| 坑是什么 | 怎么避 | 相关工具/文件 | 统一原则 |
|----------|--------|--------------|----------|
| 服务A输出 `dataInfo`，响应构建器只读 `data` | 响应构建器改为兼容读取：`data ?? dataInfo` | `src/services/ResponseBuilder.js`、`src/config/service-a-prompt.md` | 新增输出格式时必须同步检查消费端 |
| 前端状态更新只处理已有数据，新数据被忽略 | 添加 else 分支 push 新数据到数组 | `src/stores/dataStore.js` L217-L247 | 所有更新操作都应同时支持"更新已有 + 添加新" |
| 事件处理只更新已有数据 | 添加 else 分支 push 新数据 | `src/stores/appStore.js` L1504 | 事件处理器的更新逻辑应对齐状态更新逻辑 |
| 配置文件未文档化输出字段格式 | 在配置文件补充输出格式示例 | `src/config/service-b-prompt.md` | 所有输出格式约束必须与实际消费端对齐 |

## 优化建议
- 服务输出 ↔ 消费端读取的字段映射应该集中配置化，而非分散在多处
- 新增服务或修改输出格式时，必须同时修改输出规范 + 消费逻辑 + 状态更新逻辑 + 类型定义，四者缺一不可
- 建议添加自动化验证：解析配置文件中的示例字段，与消费端读取的字段做对比检查
- 重复的更新逻辑应抽取为公共函数，避免两处分别维护

## 本次可沉淀结论
- 五问法根因分析非常有效，能从现象逐层挖到本质
- 横向检查所有服务能发现系统性问题，而不只是单个bug
