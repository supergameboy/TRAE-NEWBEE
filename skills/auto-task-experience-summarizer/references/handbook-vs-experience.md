# 开发者避坑手册 vs 经验文件：区分标准详解

## 核心区分原则

| 维度 | 开发者避坑手册 | 经验文件 |
|------|----------------|----------|
| **写入条件** | 重大且具有普遍性的问题 | 已修复的单独问题、一次性bug |
| **目标读者** | 所有开发者（长期参考） | 执行相关任务的开发者（短期参考） |
| **更新频率** | 发现重大问题时 | 每次任务完成后 |
| **内容深度** | 系统性、原则性 | 具体、针对性 |

## 判断决策树

```
遇到一个问题 → 是否重大且具有普遍性？
                        ↓
                   ┌────┴────┐
                   ↓         ↓
                  是         否
                   ↓         ↓
          写入避坑手册    写入经验文件
```

## 如何判断"重大且具有普遍性"

### 三个判断标准（满足任意两个即算）

| 标准 | 说明 | 示例 |
|------|------|------|
| **影响范围广** | 影响多个模块/功能/开发者 | "前后端字段名不一致"（影响所有服务） |
| **容易重复踩** | 新开发者很可能再次犯同样错误 | "习惯性以为后端默认端口是3000" |
| **有统一原则** | 能提炼出可推广的通用原则 | "环境配置是运行前提" |

### 正面示例（应该写入避坑手册）

| 问题 | 为什么算重大 | 相关文件 |
|------|---------------|----------|
| 后端默认端口是8080不是3000 | 新开发者很容易踩坑，影响范围广 | `developer-handbook.md` 一、1 |
| 前后端必须同时启动 | 容易忘记，影响开发效率 | `developer-handbook.md` 一、2 |
| 日志函数参数必须是字符串 | 常见错误，可提炼统一原则 | `developer-handbook.md` 二、4 |
| 前后端字段名不一致 | 系统性问题，影响所有服务 | `developer-handbook.md` 四、2 |
| 外部服务输出不可信，必须容错 | 核心原则，所有服务都要遵循 | `developer-handbook.md` 六、1 |
| 状态中不应存不可序列化数据 | 架构原则，影响所有状态管理 | `developer-handbook.md` 五、3 |

### 反面示例（应该写入经验文件）

| 问题 | 为什么不算重大 | 相关文件 |
|------|----------------|----------|
| 本次修复了服务A的userInfo字段 | 单独问题，只影响服务A | `exp-20260510-bug-hunter-fractal.md` |
| 面板数据更新的具体bug修复 | 一次性问题，已修复 | `exp-20260510-bug-hunter-fractal.md` |
| 模板文件路径不存在的问题 | 特定场景，不是普遍问题 | `exp-20260512-task-planner.md` |

## 内容格式对比

### 避坑手册的章节结构

```markdown
## X、{主题}避坑

> **统一原则**：{提炼的通用原则}

### X.1 {具体问题}

**坑**：{具体描述坑}

**避**：{具体如何避免}

**查**：{如何检查是否正确}

### X.2 {下一个问题}
...
```

**特点**：
- 有章节组织，按主题分类
- 有"统一原则"（每个主题开头）
- 有"查"（如何验证）
- 长期维护，持续更新

### 经验文件的章节结构

```markdown
# 经验总结 - {日期} - {技能名称}

## 基本信息
...

## 关键步骤
...

## 坑是什么 → 怎么避 → 相关工具/文件 → 统一原则
| 坑是什么 | 怎么避 | 相关工具/文件 | 统一原则 |
|----------|--------|--------------|----------|
| ... | ... | ... | ... |

## 优化建议
...

## 本次可沉淀结论
...
```

**特点**：
- 单次任务的记录
- 四列表格格式
- 有"本次可沉淀结论"
- 归档后可合并到主题总结

## 从经验文件升级到避坑手册

如果某个经验文件中的问题被多次提到，可以考虑升级到避坑手册：

### 升级流程

1. **观察**：同一个问题在多个经验文件中出现
2. **提炼**：从具体实例中提炼通用原则
3. **分类**：确定属于避坑手册的哪个章节
4. **写入**：按避坑手册格式写入
5. **引用**：在经验文件中标记"已升级到避坑手册"

### 升级示例

**原始经验文件**（多次出现）：
```
坑：服务A输出userInfo，但响应构建器只读user
避：响应构建器改为兼容读取
```

**升级到避坑手册**：
```markdown
## 四、数据一致性与前后端同步避坑

> **统一原则**：数据变更必须双向同步——新增字段/事件/映射时，定义端和消费端必须同步更新，漏任何一环就是数据丢失

### 4.2 前后端字段名不一致

**坑**：后端返回 `user_name`，前端用 `userName`，映射遗漏导致显示undefined；服务A输出 `userInfo`，但响应构建器只读 `user`

**避**：后端使用 snake_case（数据库约定），前端使用 camelCase（JS约定）。响应到状态的映射必须完整覆盖所有字段。新增字段时同步检查前后端映射。

**查**：一致性检查工具 → 运行检查，对比前后端数据差异。
```

## 判断矩阵

| 问题特征 | 写入手册？ | 写入经验？ |
|----------|-----------|-----------|
| 只影响一个模块的具体bug | ❌ | ✅ |
| 影响所有模块的系统性问题 | ✅ | ❌ |
| 新开发者很可能再次踩坑 | ✅ | ❌ |
| 已修复的一次性问题 | ❌ | ✅ |
| 能提炼出通用原则 | ✅ | ❌ |
| 只有具体解决方案 | ❌ | ✅ |
| 需要所有开发者都知道 | ✅ | ❌ |
| 只有执行特定任务需要知道 | ❌ | ✅ |

## 混合场景处理

有些问题介于两者之间，可以：

1. **两者都写**：
   - 避坑手册：写通用原则和标准做法
   - 经验文件：写本次遇到的具体实例

2. **先写经验，观察后再升级**：
   - 先写入经验文件
   - 如果多次出现，再升级到避坑手册

**混合示例**：

避坑手册（通用原则）：
```
### 4.2 前后端字段名不一致
**避**：新增字段时同步检查前后端映射
```

经验文件（具体实例）：
```
坑：服务A输出userInfo，但响应构建器只读user
避：响应构建器改为兼容读取
相关工具/文件：ResponseBuilder.js、service-a-config.md
统一原则：新增字段时同步检查前后端映射（已在避坑手册）
```

## 何时使用 AskUserQuestion

如果不确定某个问题该写入哪里，使用 AskUserQuestion 询问用户：

**示例问题**：
```
我遇到了这个问题：{具体描述}

判断依据：
- 影响范围：{广/窄}
- 是否容易重复：{是/否}
- 能否提炼原则：{能/否}

建议：{写入避坑手册 / 写入经验文件 / 先写经验观察}

你觉得呢？
```
