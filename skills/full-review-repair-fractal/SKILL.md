# Full Review &amp; Repair Fractal - 分形式全维度审查修复流程

## 流程概述

本技能采用**分形递归思想** + **横向拆分思想**，从宏观到微观逐步深入执行完整的项目代码审查、修复和测试流程。
- **纵向**：按层级逐步细化（L0 → L1 → L2 → L3 → L4）
- **横向**：每个层级按功能模块/分类拆分，同级并行执行
- **文档**：单一总文档，层级嵌套保存
- **验证**：任务完成后进行正向和反向验证

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 | 横向拆分 |
|------|------|------|--------|----------|
| L0 | 项目级 | 整个项目概览 | 宏观 | 无（单独执行） |
| L1 | 功能模块级 | 按功能模块拆分 | 中观 | 功能模块A、功能模块B、功能模块C... |
| L2 | 子模块级 | 功能模块下的子模块 | 中观 | 子模块1、子模块2、子模块3... |
| L3 | 文件级 | 单个文件/组件 | 微观 | 文件1、文件2、文件3... |
| L4 | 代码级 | 代码片段/函数 | 原子级（可选） | 代码段1、代码段2、代码段3... |

### 递归审查模式（自相似）

每个层级都遵循相同的审查模式：

```
┌─────────────────────────────────────────────────────────┐
│  层级 N 审查模式（自相似）                                │
├─────────────────────────────────────────────────────────┤
│  1. 层级 N 审查 → 2. 展示审查报告 → 3. 选择修复范围     │
│  ↓                                                         │
│  4. 执行修复 → 5. 层级 N 验证 → 6. 判断是否深入下一层级 │
│  ↓ 是（递归）                                              │
│  层级 N+1 审查（重复上述模式）                             │
└─────────────────────────────────────────────────────────┘
```

### 横向并行执行模式

```
L0（单独执行）
    ↓
L1（按功能模块拆分，多个子Agent并行）
    ├─ 功能模块A（子Agent 1）
    ├─ 功能模块B（子Agent 2）
    ├─ 功能模块C（子Agent 3）
    └─ ...
    ↓
总结、检验、修改总文档
    ↓
L2（按子模块拆分，多个子Agent并行）
    ↓
总结、检验、修改总文档
    ↓
...
    ↓
总结、正向+反向验证
```

## 全维度评估要点

### 1. 前端评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **代码规范** | 代码风格、命名规范、注释完整性、模块化程度 |
| **渲染性能** | 页面加载速度、DOM操作效率、内存占用、动画流畅度 |
| **交互逻辑** | 事件处理、状态管理、数据流、用户操作反馈 |
| **UI还原度** | 设计稿匹配度、像素级精度、视觉一致性 |
| **兼容性** | 浏览器兼容性、设备适配、响应式布局 |

### 2. 后端评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **接口规范** | RESTful设计、API文档、参数验证、响应格式 |
| **业务逻辑** | 正确性、完整性、边界条件处理、业务规则实现 |
| **异常处理** | 错误捕获、错误信息、日志记录、优雅降级 |
| **安全防护** | 输入验证、SQL注入防护、XSS防护、认证授权 |
| **代码可读性** | 代码结构、命名规范、注释、可维护性 |

### 3. 数据库评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **SQL规范** | SQL语法、命名规范、格式化、注释 |
| **索引优化** | 索引设计、索引使用、索引维护、查询优化 |
| **事务处理** | 事务边界、隔离级别、并发控制、回滚机制 |
| **数据一致性** | 约束设计、数据校验、同步机制、冗余控制 |
| **查询性能** | 执行计划、慢查询、查询优化、连接效率 |

### 4. UI评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **布局合理性** | 信息架构、空间利用、视觉层次、对齐规范 |
| **视觉效果** | 色彩搭配、字体选择、图标设计、动效设计 |
| **用户体验** | 操作流程、反馈机制、错误提示、易用性 |
| **响应式适配** | 多设备适配、断点设计、内容重排、交互适配 |

### 5. 提示词工程评估维度（如有）
| 类别 | 详细评估要点 |
|------|-------------|
| **指令精准度** | 目标明确性、要求清晰度、任务界定 |
| **逻辑完整性** | 步骤完整性、条件覆盖、边界处理 |
| **输出可控性** | 格式规范、内容约束、质量标准 |
| **场景适配性** | 上下文理解、角色定位、场景适配 |

## 详细步骤

### 第一阶段：L0 项目级审查（宏观）- 单独执行

#### 1.1 创建任务总文档
创建单一总文档，路径：`docs/review/full-review-{YYYYMMDD}.md`

#### 1.2 项目概览审查
- 读取项目架构文档
- 检查项目目录结构
- 识别主要技术栈
- 评估项目整体健康度

#### 1.3 展示 L0 审查报告
包含：
- 项目基本信息
- 技术栈清单
- 模块划分概览
- 高优先级问题标记
- 各维度初步评分

#### 1.4 推荐 L1 横向拆分方案
向用户推荐 L1 层级的横向拆分方案（按功能模块）：
| 序号 | L1 功能模块 | 说明 | 建议优先级 |
|------|------------|------|-----------|
| 1 | 功能模块A | [说明] | 高 |
| 2 | 功能模块B | [说明] | 中 |
| 3 | 功能模块C | [说明] | 中 |

**重要**：必须使用 AskUserQuestion 工具询问用户确认/调整拆分方案！

#### 1.5 写入 L0 内容到总文档
将 L0 所有内容写入总文档。

#### 1.6 进入 L1 阶段

### 第二阶段：L1 功能模块级审查（中观）- 同级并行

#### 2.1 L1 横向并行执行
根据用户确认的 L1 拆分方案，调用多个子Agent并行执行：
- 子Agent 1：功能模块A
- 子Agent 2：功能模块B
- 子Agent 3：功能模块C
- ...

#### 2.2 每个 L1 模块的审查流程（自相似）

```
┌─────────────────────────────────────────────────────┐
│ L1 模块审查（应用自相似模式）                        │
├─────────────────────────────────────────────────────┤
│ 1. L1 模块审查 → 2. 展示 L1 报告 → 3. 选择修复问题 │
│ ↓                                                     │
│ 4. 执行 L1 修复 → 5. L1 验证 → 6. 推荐 L2 拆分方案 │
│ ↓ 询问用户确认                                        │
│ 7. 写入总文档 → 8. 是否深入 L2？                      │
└─────────────────────────────────────────────────────┘
```

#### 2.3 L1 审查内容
- 模块架构概览
- 主要功能识别

#### 2.4 推荐 L2 横向拆分方案

**重要**：必须使用 AskUserQuestion 工具询问用户确认/调整拆分方案！

#### 2.5 汇总所有 L1 模块结果
主Agent整合所有子Agent的 L1 结果，总结、检验、修改总文档。

#### 2.6 进入 L2 阶段（或跳过）

### 第三阶段：L2 子模块级审查（中观）- 同级并行

#### 3.1 L2 横向并行执行
对于每个 L1 模块下的 L2 子模块，调用多个子Agent并行执行。

#### 3.2 L2 审查流程（继续自相似模式）
同 L1 模式，推荐 L3 横向拆分方案（按文件）。

#### 3.3 汇总所有 L2 子模块结果
主Agent整合所有子Agent的 L2 结果，总结、检验、修改总文档。

---

### 第四阶段：L3 文件级审查（微观）- 同级并行

#### 4.1 L3 横向并行执行
对于每个 L2 子模块下的 L3 文件，调用多个子Agent并行执行。

#### 4.2 L3 审查流程（继续自相似模式）
同 L1 模式，可选推荐 L4 横向拆分方案（按代码段）。

#### 4.3 汇总所有 L3 文件结果
主Agent整合所有子Agent的 L3 结果，总结、检验、修改总文档。

---

### 第五阶段：L4 代码级审查（原子级）- 可选深度递归

#### 5.1 L4 横向并行执行（可选）
对于每个 L3 文件下的 L4 代码段，调用多个子Agent并行执行。

#### 5.2 L4 审查流程（继续自相似模式）
同 L1 模式，递归终止。

---

### 第六阶段：总结与验证

#### 6.1 正向验证（任务 → 文档）
逐条检查每条审查/修复，确保都有明确的文档依据：
- 文档名称
- 章节/行号
- 具体引用内容

#### 6.2 反向验证（文档 → 任务）
遍历所有文档，确保文档中的每个要求都在任务中体现：
- 建立文档内容清单
- 与任务列表逐项比对
- 标记遗漏项
- 补充缺失任务

#### 6.3 正确性验证
检查每个层级的分析/设计是否正确：
- L0 正确性
- L1 正确性
- L2 正确性
- ...

#### 6.4 一致性验证
确保各层级之间的内容一致：
- L0 → L1 一致性
- L1 → L2 一致性
- ...

#### 6.5 全局整合测试（如适用）
- 后端整合验证
- 前端整合验证
- 展示最终测试报告

#### 6.6 写入验证结果到总文档
将所有验证结果写入总文档。

## 颗粒度控制规则

### 递归深度控制
- **默认深度**：L0 → L1 → L2 → L3（建议停止在 L3）
- **可选深度**：L0 → L1 → L2 → L3 → L4（用于关键模块）
- **用户选择**：每个层级都询问用户是否继续深入

### 严重程度优先级
- **高优先级**：必须修复，阻塞流程
- **中优先级**：建议修复，可延后但不建议
- **低优先级**：可选修复，不影响功能

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似审查模式
- **必须**在每个层级开始时推荐横向拆分方案，使用 AskUserQuestion 询问用户确认
- **必须**调用多个子Agent并行执行同级任务
- **必须**每个层级全部完成后，总结、检验、修改总文档
- **必须**任务完成后进行正向+反向+正确性+一致性验证
- **必须**使用单一总文档，所有层级内容嵌套保存
- **不要**删除代码，而是注释掉
- 完成的工作写到 docs/achievement/achievement-{日期}.md 中
- 未完成的工作写到 docs/todo.md 中
- 建议使用Search Agent进行项目代码搜索，以获取相关代码信息
- 建议配合brainstorm及其他技能使用，以获得更全面的评估结果

### 子Agent规则
- 调用子Agent时，必须明确告诉子Agent的身份
- 子Agent只能申请测试，禁止独自测试（测试Agent除外）
- 子Agent完成后，将结果返回给主Agent
- 让子Agent复述上述规则

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L4 代码级（最细颗粒度）
- 当前层级无问题需要修复
- 用户主动要求停止

## 何时使用

- 项目阶段性审查
- 新功能开发完成后
- 修复bug后
- 用户要求进行全面代码审查和测试时
- 需要深度审查特定模块时
- 项目重构前的全面评估
- 用户要求进行全面项目评估、质量审查、审计、检查、检视、评审或类似评估时

## 注意事项

- 每个层级都要推荐清晰的横向拆分方案，让用户了解各种可能性
- 给予用户充分的选择权：拆分方案、修复哪些、是否深入
- 同级任务并行执行，提高效率
- 递归过程中保持上下文连贯性
- 单一总文档，层级嵌套保存，便于追溯
- 验证阶段要细致，确保完整性、正确性、一致性
- 前端测试需要启动前后端服务器
- 注意浏览器控制台消息
- 注意后台终端消息
