# Full Review &amp; Repair Fractal - 分形式全维度审查修复流程

## 流程概述

本技能采用**分形递归思想**，从宏观到微观逐步深入执行完整的项目代码审查、修复和测试流程。每个审查层级都遵循相同的自相似模式，结合全维度评估要点，确保审查的全面性和深度。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 |
|------|------|------|--------|
| L0 | 项目级 | 整个项目概览 | 宏观 |
| L1 | 领域级 | 后端/前端/数据库/UI | 中观 |
| L2 | 模块级 | 功能模块/组件组 | 中观 |
| L3 | 文件级 | 单个文件/组件 | 微观 |
| L4 | 代码级 | 代码片段/函数 | 原子级（可选） |

### 递归审查模式（自相似）

每个层级都遵循相同的审查模式：

```
┌─────────────────────────────────────────────────────────┐
│  层级 N 审查模式（自相似）                                │
├─────────────────────────────────────────────────────────┤
│  1. 层级 N 审查 → 2. 展示审查报告 → 3. 选择修复范围     │
│  ↓                                                         │
│  4. 执行修复 → 5. 层级 N 验证 → 6. 判断是否深入下一层级 │
│  ↓ 是（递归）                                              │
│  层级 N+1 审查（重复上述模式）                             │
└─────────────────────────────────────────────────────────┘
```

## 全维度评估要点

### 1. 前端评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **代码规范** | 代码风格、命名规范、注释完整性、模块化程度 |
| **渲染性能** | 页面加载速度、DOM操作效率、内存占用、动画流畅度 |
| **交互逻辑** | 事件处理、状态管理、数据流、用户操作反馈 |
| **UI还原度** | 设计稿匹配度、像素级精度、视觉一致性 |
| **兼容性** | 浏览器兼容性、设备适配、响应式布局 |

### 2. 后端评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **接口规范** | RESTful设计、API文档、参数验证、响应格式 |
| **业务逻辑** | 正确性、完整性、边界条件处理、业务规则实现 |
| **异常处理** | 错误捕获、错误信息、日志记录、优雅降级 |
| **安全防护** | 输入验证、SQL注入防护、XSS防护、认证授权 |
| **代码可读性** | 代码结构、命名规范、注释、可维护性 |

### 3. 数据库评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **SQL规范** | SQL语法、命名规范、格式化、注释 |
| **索引优化** | 索引设计、索引使用、索引维护、查询优化 |
| **事务处理** | 事务边界、隔离级别、并发控制、回滚机制 |
| **数据一致性** | 约束设计、数据校验、同步机制、冗余控制 |
| **查询性能** | 执行计划、慢查询、查询优化、连接效率 |

### 4. UI评估维度
| 类别 | 详细评估要点 |
|------|-------------|
| **布局合理性** | 信息架构、空间利用、视觉层次、对齐规范 |
| **视觉效果** | 色彩搭配、字体选择、图标设计、动效设计 |
| **用户体验** | 操作流程、反馈机制、错误提示、易用性 |
| **响应式适配** | 多设备适配、断点设计、内容重排、交互适配 |

### 5. 提示词工程评估维度（如有）
| 类别 | 详细评估要点 |
|------|-------------|
| **指令精准度** | 目标明确性、要求清晰度、任务界定 |
| **逻辑完整性** | 步骤完整性、条件覆盖、边界处理 |
| **输出可控性** | 格式规范、内容约束、质量标准 |
| **场景适配性** | 上下文理解、角色定位、场景适配 |

## 详细步骤

### 第一阶段：L0 项目级审查（宏观）

#### 1.1 项目概览审查
- 读取项目架构文档
- 检查项目目录结构
- 识别主要技术栈
- 评估项目整体健康度

#### 1.2 展示 L0 审查报告
包含：
- 项目基本信息
- 技术栈清单
- 模块划分概览
- 高优先级问题标记
- 各维度初步评分

#### 1.3 询问是否进入 L1 领域级审查

### 第二阶段：L1 领域级审查（中观）- 递归开始

#### L1 审查领域划分
- **领域 A**：后端（包含数据库）
- **领域 B**：前端（包含 UI）
- **领域 C**：提示词工程（如有）

## 颗粒度控制规则

### 递归深度控制
- **默认深度**：L0 → L1 → L2 → L3（建议停止在 L3）
- **可选深度**：L0 → L1 → L2 → L3 → L4（用于关键模块）
- **用户选择**：每个层级都询问用户是否继续深入

### 严重程度优先级
- **高优先级**：必须修复，阻塞流程
- **中优先级**：建议修复，可延后但不建议
- **低优先级**：可选修复，不影响功能

## 关键规则

### 主Agent规则
- 严格按照分形层级逐步推进，每层级都应用自相似审查模式
- 每个层级审查都应用对应的全维度评估要点
- 每个层级审查后向用户展示报告，包含维度评分
- 每个层级修复后进行该层级的验证
- 每个层级验证后询问用户是否深入下一层级（递归决策点）
- 所有层级完成后进行全局整合测试
- 测试前统一启动服务器，测试Agent禁止直接启动服务器
- 不要删除代码，而是注释掉
- 完成的工作写到 docs/achievement/achievement-{日期}.md 中
- 未完成的工作写到 docs/todo.md 中
- 建议使用Search Agent进行项目代码搜索，以获取相关代码信息
- 建议配合brainstorm及其他技能使用，以获得更全面的评估结果

### 子Agent规则
- 调用子Agent时，必须明确告诉子Agent的身份
- 子Agent只能申请测试，禁止独自测试（测试Agent除外）
- 在 docs/TestApplication.md 中写需要测试的内容
- 让子Agent复述上述规则

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L4 代码级（最细颗粒度）
- 当前层级无问题需要修复
- 用户主动要求停止

## 何时使用

- 项目阶段性审查
- 新功能开发完成后
- 修复bug后
- 用户要求进行全面代码审查和测试时
- 需要深度审查特定模块时
- 项目重构前的全面评估
- 用户要求进行全面项目评估、质量审查、审计、检查、检视、评审或类似评估时

## 注意事项

- 每个层级都要展示清晰的报告，包含维度评分，让用户了解当前进度
- 给予用户充分的选择权：修复哪些、是否深入
- 递归过程中保持上下文连贯性
- 记录每个层级的审查结果，最终汇总成完整报告
- 前端测试需要启动前后端服务器
- 注意浏览器控制台消息
- 注意后台终端消息
