# Requirements Fractal - 分形式需求分析技能

## 技能概述

本技能采用**分形递归思想**，从用户的初步想法开始，通过多层级、逐步细化的方式将概念转化为可落地的需求文档。每个决策点、分叉点、需要确认的地方都必须询问用户，每一步需求内容都会保存到文档，防止信息丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 |
|------|------|------|--------|
| L0 | 原始想法 | 用户的初步概念、想法 | 概念级 |
| L1 | 需求分类 | 功能/非功能/数据/其他需求 | 分类级 |
| L2 | 需求清单 | 具体需求条目列表 | 条目级 |
| L3 | 详细描述 | 每个需求的详细说明 | 详细级 |
| L4 | 验收条件 | 每个需求的验收标准 | 验收级（可选） |
| L5 | 测试用例 | 对应需求的测试用例 | 测试级（可选） |

### 递归分析模式（自相似）

每个层级都遵循相同的分析模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 分析模式（自相似）                                      │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 需求分析 → 2. 识别决策点/分叉点/需要确认点          │
│  ↓                                                              │
│  3. 询问用户确认 → 4. 记录需求内容 → 5. 保存当前需求文档       │
│  ↓                                                              │
│  6. 判断是否深入下一层级（递归决策点）                          │
│  ↓ 是（递归）                                                   │
│  层级 N+1 分析（重复上述模式）                                  │
└───────────────────────────────────────────────────────────────┘
```

## 详细步骤

### 第一阶段：L0 原始想法（概念级）

#### 1.1 收集用户想法
- 请用户描述初步想法或概念
- 记录用户的原始描述
- 确认理解是否正确

#### 1.2 识别 L0 决策点
- 这个想法想要解决什么问题？
- 目标用户是谁？
- 核心价值是什么？
- 期望达到什么效果？
- 时间预期是什么？

#### 1.3 询问用户确认
使用 AskUserQuestion 工具询问用户关于 L0 决策点的信息。

#### 1.4 保存 L0 需求文档
保存到：`docs/requirements/req-fractal-{YYYYMMDD}-L0.md`

#### 1.5 询问是否进入 L1 需求分类

### 第二阶段：L1 需求分类（分类级）- 递归开始

#### L1 递归分析流程（应用自相似模式）

```
┌─────────────────────────────────────────────────────┐
│ L1 需求分类（应用自相似模式）                        │
├─────────────────────────────────────────────────────┤
│ 1. L1 需求分类 → 2. 识别决策点 → 3. 询问用户 → 4. 保存 │
│ ↓                                                     │
│ 5. 是否深入 L2？ → 是则递归进入 L2 需求清单          │
└─────────────────────────────────────────────────────┘
```

#### 2.1 需求分类分析

##### L1 需求分类内容
- **功能需求**：系统需要做什么
- **非功能需求**：性能、安全、可用性等
- **数据需求**：数据存储、数据格式等
- **其他需求**：部署、运维、文档等

##### L1 需求分类决策点
| 决策点 | 说明 |
|--------|------|
| 功能需求范围 | 需要哪些功能 |
| 非功能需求重点 | 性能/安全/可用性优先级 |
| 数据需求范围 | 需要存储哪些数据 |
| 其他需求 | 部署/运维/文档要求 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 2.2 保存 L1 需求文档
保存到：`docs/requirements/req-fractal-{YYYYMMDD}-L1.md`

#### 2.3 询问是否进入 L2 需求清单

### 第三阶段：L2 需求清单（条目级）- 递归深入

#### L2 递归分析流程（继续应用自相似模式）

```
┌─────────────────────────────────────────────────────┐
│ L2 需求清单（应用自相似模式）                        │
├─────────────────────────────────────────────────────┤
│ 1. L2 需求清单 → 2. 识别决策点 → 3. 询问用户 → 4. 保存 │
│ ↓                                                     │
│ 5. 是否深入 L3？ → 是则递归进入 L3 详细描述          │

```

#### 3.1 需求清单分析

##### L2 需求清单内容
- 功能需求清单：列出所有功能条目
- 非功能需求清单：列出所有非功能条目
- 数据需求清单：列出所有数据条目
- 优先级排序：高/中/低优先级

##### L2 需求清单决策点
| 决策点 | 说明 |
|--------|------|
| 功能需求清单 | 具体功能条目列表 |
| 非功能需求清单 | 具体非功能条目列表 |
| 数据需求清单 | 具体数据条目列表 |

| MVP 范围 | 最小可行产品包含哪些 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 3.2 保存 L2 需求文档
保存到：`docs/requirements/req-fractal-{YYYYMMDD}-L2.md`

#### 3.3 询问是否进入 L3 详细描述

### 第四阶段：L3 详细描述（详细级）- 继续递归

```
┌─────────────────────────────────────────────────────┐
│ L3 详细描述（应用自相似模式）                        │
```

#### 4.1 详细描述分析

##### L3 详细描述内容
对 L2 的每个需求条目进行详细描述：
- 需求描述：详细说明需求内容
- 输入输出：输入参数、输出结果
- 业务规则：相关的业务规则

##### L3 详细描述决策点

| 决策点 | 说明 |
|--------|------|
| 需求描述 | 详细描述是否正确 |
| 输入输出 | 输入输出是否完整 |
| 业务规则 | 业务规则是否准确 |
| 依赖关系 | 依赖关系是否正确 |
| 异常情况 | 异常情况是否覆盖 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

## 文档保存规范

### 每层级文档结构

```markdown
# 分形式需求分析 - 层级 [L0-L5] - [名称]

## 1. 需求概述
- 分析时间：[时间]
- 分析层级：[L0-L5]
- 前置分析：[上一层级文档链接]
- 需求目标：[需求简述]

## 2. 需求内容
[具体需求内容]

## 3. 决策记录
| 序号 | 决策点 | 用户选择/确认 | 决策时间 |
|------|--------|--------------|---------|
| 1 | [决策点] | [用户选择] | [时间] |
| 2 | [决策点] | [用户选择] | [时间] |

## 4. 分叉点/需要确认点记录
[记录所有需要用户决策的点]

## 5. 下一层级计划
- [ ] 建议进入层级 [Lx]
- [ ] 原因：[说明原因]
```

### 文档保存时机
- **每个层级完成后**：立即保存当前层级的需求文档
- **每次用户确认后**：立即更新文档中的决策记录
- **发现新的分叉点时**：立即记录到文档中
- **递归进入下一层级前**：确保当前层级文档已保存

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似分析模式
- **必须**在每个决策点、分叉点、需要确认的地方使用 AskUserQuestion 工具询问用户
- **必须**在每一步需求分析完成后立即保存文档，防止信息丢失
- **必须**记录所有用户决策，保存到需求文档的决策记录中
- **必须**在每个层级后询问用户是否深入下一层级（递归决策点）
- **不要**自作主张做决策，所有需求内容都必须询问用户确认
- **不要**跳过层级，必须按照 L0→L1→L2→L3→L4→L5 的顺序推进
- 完成的需求分析工作写到 docs/achievement/achievement-{日期}.md 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 原始想法确认
2. 需求分类选择
3. 需求清单确认
4. 需求详细描述确认
5. 验收条件确认
6. 测试用例确认
7. 任何有多种可能的情况
8. 任何不确定用户想法的情况
9. 任何需要用户确认的需求内容

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L5 测试用例（测试级）
- 用户主动要求停止
- 需求已经细化到可落地程度，无需继续深入
- 用户已经确认所有需求内容

## 何时使用

- 用户有初步想法需要细化时
- 新项目启动前的需求分析阶段
- 新功能开发前的需求分析阶段
- 需要系统化梳理需求时
- 需要明确需求范围时
- 需要制定需求验收标准时

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种可能性
- 给予用户充分的选择权，不要预设答案
- 递归过程中保持上下文连贯性
- **每一步都要保存文档**，防止信息丢失
- 记录所有需求分析过程，便于后续追溯
- 如果有多个可能的需求方向，要询问用户先确认哪个
- 如果遇到分叉点、需要决策或确认的地方，**必须**使用 AskUserQuestion 工具问用户，**不要**自作主张
