# chat-to-skill

将多轮聊天记录提炼为可复用 Skill 资产，而不是普通总结文档。先做严格脱敏，再抽取稳定模式，最后组装完整技能包。

## 适用范围

在以下场景触发本技能：

- 用户希望把一份或一组聊天记录沉淀为新技能
- 用户希望把零散对话中的方法论抽取为 `SKILL.md`
- 用户希望从 `chat/` 一类目录生成可复用技能包
- 用户明确要求对聊天内容先脱敏，再生成技能
- 用户希望将“长期有效的操作方式”从真实项目对话中剥离出来

不要把本技能用于以下场景：

- 普通会议纪要、日报、项目复盘
- 一次性 bug 修复总结
- 只需要摘要，不需要形成 Skill 包
- 原始材料敏感但用户不允许做重写式脱敏

## 核心原则

始终遵守以下原则：

1. 先脱敏，再提炼
2. 先抽象，再组装
3. 先保留方法论，再删除一次性噪音
4. 先形成中间 outline，再写最终 Skill 文件
5. 绝不把真实项目身份信息带入技能资产

## 两阶段工作流

### 阶段一：聊天提炼

按顺序执行：

1. 读取输入源
   - 支持单个 Markdown 聊天文件
   - 支持聊天目录
   - 支持多个文件合并分析
2. 识别对话轮次
   - 优先识别 `## User` / `## Assistant`
   - 兼容普通标题、列表、代码块和命令片段
3. 执行严格脱敏
   - 替换项目名、仓库名、工作区名
   - 替换本地绝对路径、用户名目录、端口号
   - 替换文件名、文档名、素材名、品牌名、业务名
   - 改写可追溯示例，不直接做原文打码展示
4. 抽取稳定模式
   - 任务目标
   - 触发短语
   - 关键步骤
   - AskUserQuestion 决策点
   - 成功校验动作
   - 常见坑点
   - 协作接口
5. 清理噪音
   - 偶发失败
   - 临时路径
   - 一次性端口
   - 与长期方法论无关的上下文
6. 产出中间 outline

### 阶段二：技能组装

按顺序执行：

1. 根据 outline 生成 frontmatter
2. 生成精简的 `SKILL.md`
3. 把长说明拆到 `references/`
4. 用通用示例填充 `examples/`
5. 用脚本承载可重复的抽取动作
6. 更新技能系统总览
7. 做结构、触发、脱敏和脚本校验

## 输入判断

在处理前，先判断输入是否满足生成技能的条件。

### 可直接处理

- 至少包含一个较完整的问题到结果闭环
- 能看到重复性的工作模式
- 能识别出稳定约束或明确决策点
- 能从聊天中抽取出一个清晰的技能目标

### 需要先补充

- 只有零散指令，没有上下文
- 全是一次性修复，没有共性
- 材料高度敏感且没有可改写空间
- 用户只想做知识摘要，不想生成技能包

如果材料不够，先说明缺口，再请求用户补充更完整的聊天片段或更明确的目标边界。

## 抽取方法

把聊天内容拆成三类：

### 1. 可沉淀为技能

优先抽取以下内容：

- 高频任务目标
- 反复出现的步骤顺序
- 总是需要确认的决策点
- 有明确前后依赖的流程
- 验证通过的统一做法
- 适合写成规则、清单、脚本的动作

### 2. 可沉淀为参考文档

把以下内容下沉到 `references/`：

- 详细的识别模式
- 脱敏策略
- 分类标准
- 校验清单
- 边界条件

### 3. 必须剔除

删除以下内容：

- 真实项目身份信息
- 真实业务名、产品名、品牌名
- 本地路径、盘符、用户名
- 一次性报错日志
- 一次性环境细节
- 与长期复用无关的情绪化对话

## 脱敏规范

执行脱敏时，优先采用“语义改写”，不要仅做机械替换。

### 推荐改写

- 把真实项目名改为“目标项目”“示例项目”“当前仓库”
- 把真实路径改为 `workspace/chat/source.md`
- 把真实文件名改为“示例聊天文件”“结构化输出样例”
- 把真实业务领域改为“目标业务场景”
- 把真实品牌名改为“第三方平台”“目标工具”

### 保留边界

可以保留：

- Markdown、Python、JSON 这类技术类型
- `SKILL.md`、`references/`、`examples/`、`scripts/` 这类技能结构
- “提取”“校验”“脱敏”“组装”这类通用动作

不要保留：

- 能直接搜索回原项目的名词
- 具有唯一性的文档标题
- 与具体客户或业务强绑定的实体名

## 产物结构

默认生成以下结构：

```text
skills/chat-to-skill/
├── SKILL.md
├── references/
│   ├── summary-patterns.md
│   └── extraction-checklist.md
├── examples/
│   ├── sample-chat.md
│   └── sample-output.md
└── scripts/
    └── extract_skill_outline.py
```

### 文件职责

- `SKILL.md`
  承载触发条件、主流程、抽取原则、资源索引和协作接口
- `references/summary-patterns.md`
  承载模式识别、噪音剔除和脱敏方法
- `references/extraction-checklist.md`
  承载输入检查、产物检查和脱敏检查
- `examples/sample-chat.md`
  承载重写后的通用聊天示例
- `examples/sample-output.md`
  承载结构化 outline 样例
- `scripts/extract_skill_outline.py`
  承载聊天轮次识别和 outline 生成

## 验证步骤

完成技能后，依次验证：

1. 检查目录结构是否完整
2. 检查 frontmatter 是否包含强触发词
3. 检查正文是否使用指令式表达
4. 检查 `references/`、`examples/`、`scripts/` 是否都被引用
5. 检查示例是否为通用重写版本
6. 检查是否仍残留真实项目信息
7. 运行脚本验证中间 outline 是否可生成
8. 更新技能系统总览并确认定位一致

## 常见失误

避免以下失误：

- 直接把原聊天拼成 `SKILL.md`
- 在示例里残留原项目路径或文档标题
- 把一次性 bug 修复过程误写成长期规则
- 把所有内容都塞进 `SKILL.md`，不做渐进披露
- 只写文档，不提供最小可复用脚本
- 只做字符串替换，不做语义脱敏

## 资源入口

需要详细规则时，优先读取：

- `references/summary-patterns.md`
- `references/extraction-checklist.md`

需要理解输入输出样式时，优先读取：

- `examples/sample-chat.md`
- `examples/sample-output.md`

需要执行初步抽取时，优先读取或运行：

- `scripts/extract_skill_outline.py`

## 实施建议

在真实任务中，建议按以下顺序使用本技能：

1. 确认输入边界和脱敏级别
2. 读取聊天记录
3. 运行或模拟 outline 抽取
4. 审查中间产物是否仍带有身份信息
5. 写出 Skill 文件和配套资源
6. 校验并更新技能系统文档

## 技能协作接口

### 在技能体系中的定位

```text
[chat logs / chat directory] → [chat-to-skill] → [new skill package]
                                   ↓
                         [skill-development rules]
                                   ↓
                     [SKILL-SYSTEM-OVERVIEW update]
```

**本角色**：将真实聊天记录转化为匿名化、可复用技能资产的元技能。聚焦“从对话到技能”的抽取和组装，不参与具体业务执行。

### 上游输入

| 来源 | 输入内容 | 使用方式 |
|------|----------|----------|
| 用户直接输入 | 聊天文件、聊天目录、脱敏要求、命名要求 | 直接启动技能生成 |
| `skill-development` | 技能结构规范、frontmatter 规范、渐进披露原则 | 作为生成约束 |
| 现有聊天素材 | 多轮对话、修复记录、决策过程、验证动作 | 作为抽取源 |

### 下游输出

| 输出内容 | 消费者 | 使用方式 |
|----------|--------|----------|
| 新 `SKILL.md` | 技能系统 | 自动发现与触发 |
| `references/` 文档 | 后续 TRAE | 按需加载细节 |
| `examples/` 示例 | 后续 TRAE / 用户 | 理解输入输出风格 |
| 抽取脚本 | 后续 TRAE | 生成中间 outline |
| 系统总览更新 | 技能维护者 | 保持技能索引一致 |

### 协作协议

#### ← `skill-development`

- 继承技能目录结构规范
- 继承 frontmatter 和触发词设计要求
- 继承渐进披露写法

#### → 技能系统

- 将聊天方法论输出为标准技能目录
- 将长文拆分到 `references/`
- 将通用样例写入 `examples/`

#### ↔ 脱敏要求

- 在所有阶段优先执行严格脱敏
- 发现可追溯信息时，先修正再继续

### 协作约束

- 不把原始项目身份信息写入任何最终产物
- 不把一次性上下文冒充长期规则
- 不跳过中间 outline 直接生成最终技能
- Search Agent 只用于搜索，不用于写文件
