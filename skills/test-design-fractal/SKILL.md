# Test Design Fractal - 分形式测试用例设计技能

## 技能概述

本技能采用**分形递归思想**，从测试目标开始，通过多层级、逐步细化的方式设计测试用例。每个决策点、分叉点、需要确认的地方都必须询问用户，每一步测试设计内容都会保存到文档，防止信息丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 |
|------|------|------|--------|
| L0 | 测试目标 | 测什么、为什么测 | 目标级 |
| L1 | 测试类型 | 功能/性能/安全/其他 | 类型级 |
| L2 | 测试范围 | 测哪些模块/功能 | 范围级 |
| L3 | 测试场景 | 正常/异常/边界场景 | 场景级 |
| L4 | 测试用例 | 具体测试步骤、数据、预期 | 用例级 |

### 递归设计模式（自相似）

每个层级都遵循相同的设计模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 设计模式（自相似）                                      │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 测试设计 → 2. 识别决策点/分叉点/需要确认点          │
│  ↓                                                              │
│  3. 询问用户确认 → 4. 记录设计内容 → 5. 保存当前设计文档       │
│  ↓                                                              │
│  6. 判断是否深入下一层级（递归决策点）                          │
│  ↓ 是（递归）                                                   │
│  层级 N+1 设计（重复上述模式）                                  │
└───────────────────────────────────────────────────────────────┘
```

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进
- **必须**在每个决策点使用 AskUserQuestion 工具询问用户
- **必须**在每一步完成后立即保存文档
- **必须**记录所有用户决策
- **不要**自作主张做决策

### 决策点识别规则
以下情况**必须**询问用户：
1. 测试目标确认
2. 测试类型选择
3. 测试范围确认
4. 测试场景选择
5. 测试用例确认
6. 任何有多种可能的情况

### 文档保存
- 保存到：`docs/test/test-design-fractal-{YYYYMMDD}-L[0-4].md`

## 何时使用

- 新功能开发完成需要测试时
- 需求变更需要补充测试时
- 重构后需要回归测试时
- 需要系统化设计测试用例时
