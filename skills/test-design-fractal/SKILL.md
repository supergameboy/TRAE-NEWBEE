# Test Design Fractal - 分形式测试用例设计技能

## 技能概述

本技能采用**分形递归思想** + **横向拆分思想**，从测试目标开始，通过多层级、逐步细化的方式设计测试用例。
- **纵向**：按层级逐步细化（L0 → L1 → L2 → L3 → L4）
- **横向**：每个层级按功能模块拆分，同级并行执行
- **文档**：单一总文档，层级嵌套保存
- **验证**：任务完成后进行正向和反向验证

每个决策点、分叉点、需要确认的地方都必须询问用户，每一步测试设计内容都会保存到文档，防止信息丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 | 横向拆分 |
|------|------|------|--------|----------|
| L0 | 测试目标 | 测什么、为什么测 | 目标级 | 无（单独执行） |
| L1 | 功能模块级 | 按功能模块拆分 | 类型级 | 功能模块A、功能模块B、功能模块C... |
| L2 | 测试类型级 | 功能/性能/安全/其他 | 范围级 | 功能测试、性能测试、安全测试... |
| L3 | 测试场景级 | 正常/异常/边界场景 | 场景级 | 场景1、场景2、场景3... |
| L4 | 测试用例级 | 具体测试步骤、数据、预期 | 用例级 | 用例1、用例2、用例3... |

### 递归设计模式（自相似）

每个层级都遵循相同的设计模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 测试设计模式（自相似）                                  │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 测试设计 → 2. 识别决策点/分叉点/需要确认点          │
│  ↓                                                              │
│  3. 询问用户确认 → 4. 记录测试设计 → 5. 推荐横向拆分方案       │
│  ↓ 询问用户确认（如有）                                         │
│  6. 保存当前内容到总文档 → 7. 判断是否深入下一层级（递归决策点）│
│  ↓ 是（递归）                                                   │
│  层级 N+1 测试设计（重复上述模式）                              │
└───────────────────────────────────────────────────────────────┘
```

### 横向并行执行模式

```
L0（单独执行）
    ↓
L1（按功能模块拆分，多个子Agent并行）
    ├─ 功能模块A（子Agent 1）
    ├─ 功能模块B（子Agent 2）
    ├─ 功能模块C（子Agent 3）
    └─ ...
    ↓
总结、检验、修改总文档
    ↓
L2（按测试类型拆分，多个子Agent并行）
    ↓
总结、检验、修改总文档
    ↓
...
    ↓
总结、正向+反向验证
```

# 总结与验证

## 正向验证（测试 → 文档）
逐条检查每条测试，确保都有明确的文档依据：
- [ ] 验证项1
- [ ] 验证项2

## 反向验证（文档 → 测试）
遍历所有文档，确保文档中的每个要求都在测试中体现：
- [ ] 验证项1
- [ ] 验证项2

## 正确性验证

### 文档保存时机
- **任务开始时**：创建任务总文档，路径：`docs/test/test-design-fractal-{YYYYMMDD}.md`
- **L0 完成后**：写入 L0 内容，询问 L1 横向拆分方案
- **每个 L1 模块完成后**：写入对应 L1 内容，询问 L2 横向拆分方案
- **每个层级全部完成后**：总结、检验、修改总文档
- **任务完成后**：进行正向+反向+正确性+一致性验证，写入验证结果

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似测试设计模式
- **必须**在每个层级开始时推荐横向拆分方案，使用 AskUserQuestion 询问用户确认
- **必须**调用多个子Agent并行执行同级任务
- **必须**每个层级全部完成后，总结、检验、修改总文档
- **必须**任务完成后进行正向+反向+正确性+一致性验证
- **必须**使用单一总文档，所有层级内容嵌套保存
- **必须**在每个决策点、分叉点、需要确认的地方使用 AskUserQuestion 工具询问用户
- **不要**自作主张做决策
- 完成的测试设计工作写到 `docs/achievement/achievement-{日期}.md` 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 测试目标确认
2. 测试类型选择
3. 测试范围确认
4. 测试场景选择
5. 测试用例确认
6. 横向拆分方案确认
7. 任何有多种可能的情况

### 递归终止条件
- 用户选择不继续深入下一层级
- 用户主动要求停止

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种可能性
- 每个层级都要推荐清晰的横向拆分方案
- 给予用户充分的选择权，不要预设答案
- 同级任务并行执行，提高效率
- 递归过程中保持上下文连贯性
- **每一步都要保存文档**，防止信息丢失
- 单一总文档，层级嵌套保存，便于追溯
- 记录所有测试设计过程，便于后续追溯
- 如果有多个可能的测试方向，要询问用户先尝试哪个
- 如果遇到分叉点、需要决策或确认的地方，**必须**使用 AskUserQuestion 工具问用户，**不要**自作主张
