# Bug Hunter Fractal - 分形式问题排查技能

## 技能概述

本技能采用**分形递归思想** + **横向拆分思想**，从问题现象开始，通过多层级、逐步细化的方式定位问题根源。
- **纵向**：按层级逐步细化（L0 → L1 → L2 → L3 → L4 → L5）
- **横向**：每个层级按问题领域/分类拆分，同级并行执行
- **文档**：单一总文档，层级嵌套保存
- **验证**：任务完成后进行正向和反向验证

每个决策点、分叉点、需要确认的地方都必须询问用户，每一步排查内容都会保存到文档，防止思路丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 | 横向拆分 |
|------|------|------|--------|----------|
| L0 | 现象描述 | 用户观察到的问题现象 | 现象级 | 无（单独执行） |
| L1 | 领域判断 | 按问题领域拆分 | 领域级 | 前端、后端、数据库、网络、其他... |
| L2 | 模块定位 | 具体哪个功能模块 | 模块级 | 模块1、模块2、模块3... |
| L3 | 文件定位 | 具体哪个文件 | 文件级 | 文件1、文件2、文件3... |
| L4 | 代码定位 | 具体哪段代码/函数 | 代码级（可选） | 代码段1、代码段2、代码段3... |
| L5 | 根因分析 | 问题根本原因 | 根源级（可选） | 无 |

### 递归排查模式（自相似）

每个层级都遵循相同的排查模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 排查模式（自相似）                                      │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 信息收集 → 2. 识别决策点/分叉点/需要确认点          │
│  ↓                                                              │
│  3. 询问用户确认 → 4. 记录排查过程 → 5. 推荐横向拆分方案       │
│  ↓ 询问用户确认（如有）                                         │
│  6. 保存当前内容到总文档 → 7. 判断是否深入下一层级（递归决策点）│
│  ↓ 是（递归）                                                   │
│  层级 N+1 排查（重复上述模式）                                  │
└───────────────────────────────────────────────────────────────┘
```

### 横向并行执行模式

```
L0（单独执行）
    ↓
L1（按问题领域拆分，多个子Agent并行）
    ├─ 前端（子Agent 1）
    ├─ 后端（子Agent 2）
    ├─ 数据库（子Agent 3）
    ├─ 网络（子Agent 4）
    └─ 其他（子Agent 5）
    ↓
总结、检验、修改总文档
    ↓
L2（按模块拆分，多个子Agent并行）
    ↓
总结、检验、修改总文档
    ↓
...
    ↓
总结、正向+反向验证
```

## 详细步骤

### 第一阶段：L0 现象描述（现象级）- 单独执行

#### 1.1 创建任务总文档
创建单一总文档，路径：`docs/debug/bug-hunt-{YYYYMMDD}.md`

#### 1.2 收集问题现象
- 请用户描述观察到的问题现象
- 记录用户的原始描述
- 确认理解是否正确

#### 1.3 识别 L0 决策点
- 问题是什么时候出现的？
- 问题出现的频率？
- 问题是否稳定复现？
- 问题出现前做了什么操作？
- 预期的正常行为是什么？

#### 1.4 询问用户确认
使用 AskUserQuestion 工具询问用户关于 L0 决策点的信息。

#### 1.5 推荐 L1 横向拆分方案
向用户推荐 L1 层级的横向拆分方案（按问题领域）：
| 序号 | L1 问题领域 | 说明 | 建议优先级 |
|------|------------|------|-----------|
| 1 | 前端 | 页面显示、交互、控制台错误 | 高 |
| 2 | 后端 | API响应、服务器日志、错误码 | 高 |
| 3 | 数据库 | 数据异常、查询错误、连接问题 | 中 |
| 4 | 网络 | 请求失败、超时、跨域问题 | 中 |
| 5 | 其他 | 环境问题、配置问题、依赖问题 | 低 |

**重要**：必须使用 AskUserQuestion 工具询问用户确认/调整拆分方案！

#### 1.6 写入 L0 内容到总文档
将 L0 所有内容写入总文档。

#### 1.7 询问是否进入 L1 领域判断

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似排查模式
- **必须**在每个层级开始时推荐横向拆分方案，使用 AskUserQuestion 询问用户确认
- **必须**调用多个子Agent并行执行同级任务
- **必须**每个层级全部完成后，总结、检验、修改总文档
- **必须**任务完成后进行正向+反向+正确性+一致性验证
- **必须**使用单一总文档，所有层级内容嵌套保存
- **必须**在每个决策点、分叉点、需要确认的地方使用 AskUserQuestion 工具询问用户
- **不要**自作主张做判断，所有问题定位都必须询问用户确认
- **不要**跳过层级，必须按照 L0→L1→L2→L3→L4→L5 的顺序推进
- 完成的排查工作写到 docs/achievement/achievement-{日期}.md 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 问题现象确认
2. 问题领域判断
3. 模块定位选择
4. 文件定位选择
5. 代码定位选择
6. 根因分析确认
7. 横向拆分方案确认
8. 任何有多种可能的情况
9. 任何不确定用户想法的情况
10. 任何需要用户确认的排查方向

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L5 根因分析（根源级）
- 用户主动要求停止
- 问题已经定位，无需继续深入
- 用户已经确认问题根源

## 何时使用

- 发现 bug 需要排查时
- 用户反馈问题需要定位时
- 系统出现异常需要调查时
- 性能问题需要分析时
- 任何需要从现象到根源层层定位的问题

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种可能性
- 每个层级都要推荐清晰的横向拆分方案
- 给予用户充分的选择权，不要预设答案
- 同级任务并行执行，提高效率
