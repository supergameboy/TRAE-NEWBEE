# Bug Hunter Fractal - 分形式问题排查技能

## 技能概述

本技能采用**分形递归思想**，从问题现象开始，通过多层级、逐步细化的方式定位问题根源。每个决策点、分叉点、需要确认的地方都必须询问用户，每一步排查内容都会保存到文档，防止思路丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 |
|------|------|------|--------|
| L0 | 现象描述 | 用户观察到的问题现象 | 现象级 |
| L1 | 领域判断 | 前端/后端/数据库/网络/其他 | 领域级 |
| L2 | 模块定位 | 具体哪个功能模块 | 模块级 |
| L3 | 文件定位 | 具体哪个文件 | 文件级 |
| L4 | 代码定位 | 具体哪段代码/函数 | 代码级（可选） |
| L5 | 根因分析 | 问题根本原因 | 根源级（可选） |

### 递归排查模式（自相似）

每个层级都遵循相同的排查模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 排查模式（自相似）                                      │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 信息收集 → 2. 识别决策点/分叉点/需要确认点          │
│  ↓                                                              │
│  3. 询问用户确认 → 4. 记录排查过程 → 5. 保存当前排查文档       │
│  ↓                                                              │
│  6. 判断是否深入下一层级（递归决策点）                          │
│  ↓ 是（递归）                                                   │
│  层级 N+1 排查（重复上述模式）                                  │
└───────────────────────────────────────────────────────────────┘
```

## 详细步骤

### 第一阶段：L0 现象描述（现象级）

#### 1.1 收集问题现象
- 请用户描述观察到的问题现象
- 记录用户的原始描述
- 确认理解是否正确

#### 1.2 识别 L0 决策点
- 问题是什么时候出现的？
- 问题出现的频率？
- 问题是否稳定复现？
- 问题出现前做了什么操作？
- 预期的正常行为是什么？

#### 1.3 询问用户确认
使用 AskUserQuestion 工具询问用户关于 L0 决策点的信息。

#### 1.4 保存 L0 排查文档
保存到：`docs/debug/bug-hunt-{YYYYMMDD}-L0.md`

#### 1.5 询问是否进入 L1 领域判断

### 第二阶段：L1 领域判断（领域级）- 递归开始

#### L1 递归排查流程（应用自相似模式）

```
┌─────────────────────────────────────────────────────┐
│ L1 领域判断（应用自相似模式）                        │
├─────────────────────────────────────────────────────┤
│ 1. L1 信息收集 → 2. 识别决策点 → 3. 询问用户 → 4. 保存 │
│ ↓                                                     │
│ 5. 是否深入 L2？ → 是则递归进入 L2 模块定位          │
└─────────────────────────────────────────────────────┘
```

#### 2.1 领域判断分析

##### L1 领域判断内容
- 前端表现：页面显示、交互、控制台错误
- 后端表现：API响应、服务器日志、错误码
- 数据库表现：数据异常、查询错误、连接问题
- 网络表现：请求失败、超时、跨域问题
- 其他：环境问题、配置问题、依赖问题

##### L1 领域判断决策点
| 决策点 | 可选选项 | 说明 |
|--------|---------|------|
| 问题领域 | 前端 / 后端 / 数据库 / 网络 / 其他 | 判断问题所属领域 |
| 错误来源 | 控制台错误 / 服务器日志 / 用户反馈 / 其他 | 错误信息来源 |
| 复现方式 | 稳定复现 / 偶现 / 需要特定条件 | 问题复现情况 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 2.2 领域验证（可选）
如果有多个可能的领域，询问用户先排查哪个。

#### 2.3 保存 L1 排查文档
保存到：`docs/debug/bug-hunt-{YYYYMMDD}-L1.md`

#### 2.4 询问是否进入 L2 模块定位

### 第三阶段：L2 模块定位（模块级）- 递归深入

#### L2 递归排查流程（继续应用自相似模式）

```
┌─────────────────────────────────────────────────────┐
│ L2 模块定位（应用自相似模式）                        │

└─────────────────────────────────────────────────────┘
```

#### 3.1 模块定位分析

##### L2 模块定位内容
- 根据 L1 确定的领域，列出相关模块
- 分析每个模块与问题的关联度
- 识别可能的问题模块

##### L2 模块定位决策点
| 决策点 | 说明 |
|--------|------|
| 相关模块清单 | 可能涉及的模块列表 |
| 可疑模块 | 最可能有问题的模块 |
| 排查顺序 | 先排查哪个模块 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 3.2 保存 L2 排查文档
保存到：`docs/debug/bug-hunt-{YYYYMMDD}-L2.md`

#### 3.3 询问是否进入 L3 文件定位

## 文档保存规范

### 每层级文档结构

```markdown
# 分形式问题排查 - 层级 [L0-L5] - [名称]

## 1. 排查概述
- 排查时间：[时间]
- 排查层级：[L0-L5]
- 前置排查：[上一层级文档链接]
- 问题描述：[问题简述]

## 2. 排查内容
[具体排查内容]

## 3. 决策记录
| 序号 | 决策点 | 用户选择/确认 | 决策时间 |
|------|--------|--------------|---------|
| 1 | [决策点] | [用户选择] | [时间] |
| 2 | [决策点] | [用户选择] | [时间] |

## 4. 分叉点/需要确认点记录
[记录所有需要用户决策的点]

## 5. 下一层级计划
- [ ] 建议进入层级 [Lx]
- [ ] 原因：[说明原因]
```

### 文档保存时机
- **每个层级完成后**：立即保存当前层级的排查文档
- **每次用户确认后**：立即更新文档中的决策记录
- **发现新的分叉点时**：立即记录到文档中
- **递归进入下一层级前**：确保当前层级文档已保存

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似排查模式
- **必须**在每个决策点、分叉点、需要确认的地方使用 AskUserQuestion 工具询问用户
- **必须**在每一步排查完成后立即保存文档，防止思路丢失
- **必须**记录所有用户决策，保存到排查文档的决策记录中
- **必须**在每个层级后询问用户是否深入下一层级（递归决策点）
- **不要**自作主张做判断，所有问题定位都必须询问用户确认
- **不要**跳过层级，必须按照 L0→L1→L2→L3→L4→L5 的顺序推进
- 完成的排查工作写到 docs/achievement/achievement-{日期}.md 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 问题现象确认
2. 问题领域判断
3. 模块定位选择
4. 文件定位选择
5. 代码定位选择
6. 根因分析确认
7. 任何有多种可能的情况
8. 任何不确定用户想法的情况
9. 任何需要用户确认的排查方向

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L5 根因分析（根源级）
- 用户主动要求停止
- 问题已经定位，无需继续深入
- 用户已经确认问题根源

## 何时使用

- 发现 bug 需要排查时
- 用户反馈问题需要定位时
- 系统出现异常需要调查时
- 性能问题需要分析时
- 任何需要从现象到根源层层定位的问题

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种可能性
- 给予用户充分的选择权，不要预设答案
- 递归过程中保持上下文连贯性
- **每一步都要保存文档**，防止思路丢失
- 记录所有排查过程，便于后续追溯
- 如果有多个可能的排查方向，要询问用户先尝试哪个
- 如果遇到分叉点、需要决策或确认的地方，**必须**使用 AskUserQuestion 工具问用户，**不要**自作主张
