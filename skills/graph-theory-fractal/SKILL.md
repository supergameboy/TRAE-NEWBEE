# Graph Theory Fractal - 分形式图论分析技能

## 技能概述

本技能采用**分形递归思想**结合**图论思想**，从系统/模块分析开始，通过多层级、逐步细化的方式识别节点、定义边、分析图结构、提出优化建议。每个决策点、分叉点、需要确认的地方都必须询问用户，每一步分析内容都会保存到文档，防止思路丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 |
|------|------|------|--------|
| L0 | 图主题定义 | 要分析的系统/模块是什么 | 主题级 |
| L1 | 节点识别 | 识别系统中的节点（实体、模块、组件等） | 节点级 |
| L2 | 边的定义 | 定义节点之间的关系（数据流、依赖、状态转移等） | 边级 |
| L3 | 图的分析 | 路径分析、循环检测、拓扑排序、关键路径识别 | 分析级 |
| L4 | 优化建议 | 基于图分析的优化建议 | 优化级（可选） |

### 递归分析模式（自相似）

每个层级都遵循相同的分析模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 分析模式（自相似）                                      │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 信息收集 → 2. 识别决策点/分叉点/需要确认点          │
│  ↓                                                              │
│  3. 询问用户确认 → 4. 记录分析过程 → 5. 保存当前分析文档       │
│  ↓                                                              │
│  6. 判断是否深入下一层级（递归决策点）                          │
│  ↓ 是（递归）                                                   │
│  层级 N+1 分析（重复上述模式）                                  │
└───────────────────────────────────────────────────────────────┘
```

## 详细步骤

### 第一阶段：L0 图主题定义（主题级）

#### 1.1 收集图主题
- 请用户描述要分析的系统/模块
- 记录用户的原始描述
- 确认理解是否正确

#### 1.2 识别 L0 决策点
| 决策点 | 可选选项 | 说明 |
|--------|---------|------|
| 图类型 | 数据流图 / 依赖关系图 / 状态转移图 / 调用关系图 / 其他 | 图的类型 |
| 分析目标 | 性能优化 / 架构优化 / 循环检测 / 依赖梳理 / 其他 | 分析目标 |
| 分析范围 | 单个模块 / 多个模块 / 整个系统 | 分析范围 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 1.3 保存 L0 分析文档
保存到：`docs/graph/graph-analysis-{YYYYMMDD}-L0.md`

#### 1.4 询问是否进入 L1 节点识别

#### 2.1 节点识别分析

##### L1 节点识别内容
根据图类型识别节点：
- **数据流图**：数据生产者、数据消费者、数据存储
- **依赖关系图**：模块、组件、函数、类

- **调用关系图**：函数、方法、API
- **其他**：根据图类型自定义

| 节点清单 | 列出所有识别到的节点 |
| 节点类型 | 每个节点的类型/角色 |
| 关键节点 | 哪些节点是核心/关键节点 |
| 是否有遗漏节点 | 是否需要补充节点 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 2.2 保存 L1 分析文档
保存到：`docs/graph/graph-analysis-{YYYYMMDD}-L1.md`

### 第四阶段：L3 图的分析（分析级）- 继续递归

```

└─────────────────────────────────────────────────────┘
```

#### 4.1 图的分析内容

##### L3 图分析内容
根据图类型进行分析：

**通用分析**：
- 连通性分析：图是否连通，有多少连通分量
- 路径分析：节点之间的路径、最短路径、最长路径
- 循环检测：图中是否存在循环（环）
- 度数分析：节点的入度、出度、度数分布

**数据流图**：
- 数据流路径
- 数据瓶颈识别
- 数据一致性检查

**依赖关系图**：
- 拓扑排序（如果是有向无环图）
- 循环依赖检测
- 依赖层级分析

**状态转移图**：
- 可达性分析（哪些状态可达）
- 死锁检测
- 状态转换路径

**调用关系图**：
- 调用链分析
- 调用深度分析
- 热点识别（频繁调用的函数）

##### L3 图分析决策点
| 决策点 | 说明 |
|--------|------|
| 分析项选择 | 选择要进行的分析项 |
| 关键路径识别 | 识别关键路径/关键节点 |
| 问题识别 | 识别出的问题（循环、瓶颈、死锁等） |
| 分析结果确认 | 确认分析结果是否正确 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 5.1 优化建议内容

##### L4 优化建议内容
根据 L3 的分析结果提出优化建议：

**循环问题**：
- 循环依赖的解决方案
- 循环调用的优化方案
- 状态循环的处理建议

**瓶颈问题**：
- 数据瓶颈的优化（缓存、并行处理等）
- 调用瓶颈的优化（异步、批量处理等）
- 依赖瓶颈的优化（解耦、模块化等）

**架构优化**：
- 模块化建议
- 解耦建议
- 分层建议

**性能优化**：
- 路径缩短建议
- 热点优化建议
- 资源分配优化建议

##### L4 优化建议决策点

| 优化实施计划 | 优化的实施计划 |
| 优化效果预期 | 预期的优化效果 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

### 文档保存时机
- **每个层级完成后**：立即保存当前层级的分析文档
- **每次用户确认后**：立即更新文档中的决策记录
- **发现新的分叉点时**：立即记录到文档中
- **递归进入下一层级前**：确保当前层级文档已保存

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似分析模式
- **必须**在每个决策点、分叉点、需要确认的地方使用 AskUserQuestion 工具询问用户
- **必须**在每一步分析完成后立即保存文档，防止思路丢失
- **必须**记录所有用户决策，保存到分析文档的决策记录中
- **必须**在每个层级后询问用户是否深入下一层级（递归决策点）
- **不要**自作主张做判断，所有分析都必须询问用户确认
- **不要**跳过层级，必须按照 L0→L1→L2→L3→L4 的顺序推进
- 完成的分析工作写到 docs/achievement/achievement-{日期}.md 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 图类型和分析目标确认
2. 节点清单和节点类型确认
3. 边的清单和边的类型确认
4. 分析项选择和分析结果确认
5. 优化方案选择和优先级确定
6. 任何有多种可能的情况
7. 任何不确定用户想法的情况
8. 任何需要用户确认的分析方向

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L4 优化建议（优化级）
- 用户主动要求停止
- 分析已达到目标，无需继续深入
- 用户已经确认分析结果

## 何时使用

- 需要分析数据流时
- 需要梳理依赖关系时
- 需要分析状态转移时
- 需要分析调用关系时
- 需要检测循环依赖时
- 需要进行架构优化时
- 需要进行性能优化时
- 任何需要用图论思想分析的场景

## 注意事项

- **每个决策点**都要展示清晰的选项，让用户了解各种可能性
- 给予用户充分的选择权，不要预设答案
- 递归过程中保持上下文连贯性
- **每一步都要保存文档**，防止思路丢失
- 记录所有分析过程，便于后续追溯
- 如果有多个可能的分析方向，要询问用户先尝试哪个
- 如果遇到分叉点、需要决策或确认的地方，**必须**使用 AskUserQuestion 工具问用户，**不要**自作主张
- 可以尝试用 Mermaid 语法生成图的可视化表示
