# Graph Theory Fractal - 分形式图论分析技能

## 技能概述

本技能采用**分形递归思想** + **横向拆分思想**，结合**图论思想**，从系统/模块分析开始，通过多层级、逐步细化的方式识别节点、定义边、分析图结构、提出优化建议。
- **纵向**：按层级逐步细化（L0 → L1 → L2 → L3 → L4）
- **横向**：每个层级按功能模块/分类拆分，同级并行执行
- **文档**：单一总文档，层级嵌套保存
- **验证**：任务完成后进行正向和反向验证

每个决策点、分叉点、需要确认的地方都必须询问用户，每一步分析内容都会保存到文档，防止思路丢失。

## 分形工作流程

### 分形层级定义

| 层级 | 名称 | 说明 | 颗粒度 | 横向拆分 |
|------|------|------|--------|----------|
| L0 | 图主题定义 | 要分析的系统/模块是什么 | 主题级 | 无（单独执行） |
| L1 | 功能模块级 | 按功能模块拆分 | 模块级 | 功能模块A、功能模块B、功能模块C... |
| L2 | 子模块级 | 功能模块下的子模块 | 子模块级 | 子模块1、子模块2、子模块3... |
| L3 | 节点/边级 | 具体节点和边 | 分析级 | 节点1、节点2、边1、边2... |
| L4 | 优化建议 | 基于图分析的优化建议 | 优化级（可选） | 无 |

### 递归分析模式（自相似）

每个层级都遵循相同的分析模式：

```
┌───────────────────────────────────────────────────────────────┐
│  层级 N 分析模式（自相似）                                      │
├───────────────────────────────────────────────────────────────┤
│  1. 层级 N 信息收集 → 2. 识别决策点/分叉点/需要确认点          │
│  ↓                                                              │
│  3. 询问用户确认 → 4. 记录分析过程 → 5. 推荐横向拆分方案       │
│  ↓ 询问用户确认（如有）                                         │
│  6. 保存当前内容到总文档 → 7. 判断是否深入下一层级（递归决策点）│
│  ↓ 是（递归）                                                   │
│  层级 N+1 分析（重复上述模式）                                  │
└───────────────────────────────────────────────────────────────┘
```

### 横向并行执行模式

```
L0（单独执行）
    ↓
L1（按功能模块拆分，多个子Agent并行）
    ├─ 功能模块A（子Agent 1）
    ├─ 功能模块B（子Agent 2）
    ├─ 功能模块C（子Agent 3）
    └─ ...
    ↓
总结、检验、修改总文档
    ↓
L2（按子模块拆分，多个子Agent并行）
    ↓
总结、检验、修改总文档
    ↓
...
    ↓
总结、正向+反向验证
```

### 第四阶段：L3 节点/边级分析（分析级）- 同级并行

#### 4.1 L3 横向并行执行
对于每个 L2 子模块下的 L3 节点/边，调用多个子Agent并行执行。

#### 4.2 L3 分析流程（继续自相似模式）
同 L1 模式，推荐 L4 优化建议。

#### 4.3 L3 分析内容
根据图类型进行分析：

**通用分析**：
- 连通性分析：图是否连通，有多少连通分量
- 路径分析：节点之间的路径、最短路径、最长路径
- 循环检测：图中是否存在循环（环）
- 度数分析：节点的入度、出度、度数分布

**数据流图**：
- 数据流路径
- 数据瓶颈识别
- 数据一致性检查

**依赖关系图**：
- 拓扑排序（如果是有向无环图）
- 循环依赖检测
- 依赖层级分析

**状态转移图**：
- 可达性分析（哪些状态可达）
- 死锁检测
- 状态转换路径

**调用关系图**：
- 调用链分析
- 调用深度分析
- 热点识别（频繁调用的函数）

#### 4.4 L3 决策点
| 决策点 | 说明 |
|--------|------|
| 分析项选择 | 选择要进行的分析项 |
| 关键路径识别 | 识别关键路径/关键节点 |
| 问题识别 | 识别出的问题（循环、瓶颈、死锁等） |
| 分析结果确认 | 确认分析结果是否正确 |

**重要**：以上决策点必须使用 AskUserQuestion 工具询问用户！

#### 4.5 汇总所有 L3 节点/边结果

**架构优化**：

- 分层建议

**性能优化**：
- 路径缩短建议
- 热点优化建议
- 资源分配优化建议

#### L4 决策点
| 决策点 | 说明 |
|--------|------|
| 优化优先级 | 优化项的优先级排序 |

## 关键规则

### 主Agent规则
- **必须**严格按照分形层级逐步推进，每层级都应用自相似分析模式
- **必须**在每个层级开始时推荐横向拆分方案，使用 AskUserQuestion 询问用户确认
- **必须**调用多个子Agent并行执行同级任务
- **必须**每个层级全部完成后，总结、检验、修改总文档
- **必须**任务完成后进行正向+反向+正确性+一致性验证
- **必须**使用单一总文档，所有层级内容嵌套保存
- **必须**在每个决策点、分叉点、需要确认的地方使用 AskUserQuestion 工具询问用户
- **不要**自作主张做判断，所有分析都必须询问用户确认
- **不要**跳过层级，必须按照 L0→L1→L2→L3→L4 的顺序推进
- 完成的分析工作写到 docs/achievement/achievement-{日期}.md 中

### 决策点识别规则
以下情况**必须**询问用户：
1. 图类型和分析目标确认
2. 节点清单和节点类型确认
3. 边的清单和边的类型确认
4. 分析项选择和分析结果确认
5. 优化方案选择和优先级确定
6. 横向拆分方案确认
7. 任何有多种可能的情况
8. 任何不确定用户想法的情况
9. 任何需要用户确认的分析方向

### 递归终止条件
- 用户选择不继续深入下一层级
- 已到达 L4 优化建议（优化级）
- 用户主动要求停止
- 分析已达到目标，无需继续深入
- 用户已经确认分析结果

## 何时使用

- 需要分析数据流时
- 需要梳理依赖关系时
